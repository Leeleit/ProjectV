# Философия времени компиляции: смерть `#include`, эра модулей C++26

Время компиляции — это не просто неудобство. Это **прямой враг итеративной разработки**. Если сборка движка занимает 10
минут, ты не будешь экспериментировать, рефакторить, пробовать новые идеи. Ты будешь бояться нажать «собрать».

Мы убиваем эту проблему на корню. Модули C++26 — это не просто фича, это **революция в том, как мы организуем код**.

---

## Почему `#include` убивает время сборки?

`#include` — это текстовое включение. Механически, грубо, без понимания.

```cpp
// main.cpp
#include <vector>
#include <string>
#include <algorithm>
// ... ещё 50 заголовков

// Компилятор видит:
// 1. Скопировать весь <vector> (~50 000 строк)
// 2. Скопировать весь <string> (~20 000 строк)
// 3. Скопировать весь <algorithm> (~30 000 строк)
// Итого: 100 000+ строк парсятся заново в КАЖДОМ .cpp файле.
```

**Проблемы:**

1. **Дублирование:** Один и тот же код парсится N раз.
2. **Циклические зависимости:** `A.h` включает `B.h`, `B.h` включает `A.h` — компилятор впадает в бесконечный цикл или
   требует forward declarations.
3. **Медленный парсинг:** Каждый `#include` — это открытие файла, чтение, разбор.
4. **Хрупкость:** Изменение одного заголовка вызывает перекомпиляцию **всех** `.cpp`, которые его включают.

> **Метафора:** Представь, что ты пишешь книгу. `#include` — это когда ты каждый раз переписываешь главу из другой книги
> в свою. Хочешь использовать главу «Векторы» — переписываешь её. Хочешь «Строки» — переписываешь ещё раз. В итоге твоя
> книга состоит из 90% чужих текстов, которые ты копируешь снова и снова. Безумие.

---

## Модули C++26: как это работает?

Модуль — это **бинарное представление интерфейса**. Он компилируется **один раз**, затем импортируется как готовый блок.

```cpp
// math.ixx (модуль)
export module math;

export int add(int a, int b) { return a + b; }
export int mul(int a, int b) { return a * b; }

// main.cpp
import math;  // Не текст, а бинарный интерфейс

int main() {
    return add(2, 3);  // Компилятор знает сигнатуру, не парсит тело
}
```

**Преимущества:**

1. **Одна компиляция модуля** — потом только импорт.
2. **Нет циклических зависимостей** — модули образуют DAG.
3. **Изоляция:** Модуль видит только то, что экспортирует.
4. **Скорость:** Импорт в 10–100 раз быстрее, чем `#include`.

## Разделение интерфейса и реализации

В мире `#include` интерфейс и реализация смешаны. Заголовочный файл содержит объявления **и** определения (inline
функции, шаблоны).

В мире модулей мы чётко разделяем:

```cpp
// math.ixx (интерфейс модуля)
export module math;

export int add(int a, int b);
export int mul(int a, int b);

// math.cpp (реализация модуля)
module math;

int add(int a, int b) { return a + b; }
int mul(int a, int b) { return a * b; }
```

**Почему это важно:**

- Изменение реализации не затрагивает импортёров.
- Компилятор видит интерфейс один раз, кэширует его.
- Лучшая инкапсуляция: приватные детали остаются внутри модуля.

## Быстрый билд = быстрая разработка

Скорость сборки напрямую влияет на скорость разработки. Если ты ждёшь компиляцию 5 минут, ты:

- Меньше экспериментируешь
- Боишься рефакторить
- Пишешь больше кода «наугад», потому что проверить дорого

С модулями мы сокращаем время сборки в **2–5 раз**. Это не просто удобство — это **качественный скачок в продуктивности
**.

> **Метафора:** Представь двух поваров. У одного плита греется 10 минут. Он будет готовить одно блюдо, боясь ошибиться.
> У второго плита мгновенная. Он экспериментирует, пробует новые сочетания, быстро исправляет ошибки. Кто станет лучшим
> поваром? Модули — это мгновенная плита для программиста.

## Практика: как мы переходим на модули

### 1. Постепенная миграция

Мы не переписываем всё сразу. Мы начинаем с ядра:

- Сначала `core.ixx` — базовые типы, аллокаторы, утилиты.
- Потом `math.ixx` — векторы, матрицы, кватернионы.
- Потом `ecs.ixx` — ECS‑ядро.

Каждый модуль изолирует свою область, уменьшая связность.

### 2. Совместимость с легаси

Библиотеки типа Vulkan, SDL, Flecs пока остаются с `#include`. Но наш код общается с ними через **import** своих
обёрток.

```cpp
import vulkan_wrapper;  // Наш модуль-обёртка над Vulkan‑заголовками
```

Со временем и внешние библиотеки перейдут на модули — стандарт уже готов.

### 3. Инструменты

Clang 19+ полностью поддерживает модули. CMake 3.28+ умеет с ними работать. Мы настраиваем сборку так:

- `.ixx` файлы компилируются как модули
- `.cpp` файлы импортируют модули
- Зависимости автоматически разрешаются

## `constexpr` и вычисления при компиляции

Модули — не единственный способ ускорить разработку. `constexpr` переносит вычисления из рантайма в compile‑time.

```cpp
constexpr int chunk_size = 16;
constexpr int voxels_per_chunk = chunk_size * chunk_size * chunk_size;  // Вычисляется при компиляции

// Таблица синусов для генерации ландшафта
constexpr std::array<float, 1024> generate_sin_table() {
    std::array<float, 1024> result{};
    for (size_t i = 0; i < result.size(); ++i) {
        result[i] = std::sin(2.0f * 3.141592f * i / result.size());
    }
    return result;
}
constexpr auto sin_table = generate_sin_table();  // Вычисляется один раз при компиляции
```

**Что это даёт:**

- Нет вычислений в рантайме
- Данные готовы сразу в бинарнике
- Возможность использовать сложную логику для инициализации констант

## Итог: время компиляции как метрика

Мы отслеживаем время сборки так же, как FPS. Если оно растёт — ищем причину:

- Циклические `#include`
- Гигантские заголовки
- Отсутствие forward declarations
- Неиспользование модулей

Правила:

1. **Модули везде, где возможно.** `#include` только для легаси.
2. **`constexpr` для compile‑time вычислений.** Зачем считать в рантайме, если можно при компиляции?
3. **Инкрементальная сборка всегда работает.** Если изменение в одном модуле пересобирает весь проект — архитектура
   сломана.
4. **Быстрая сборка — это фича.** Так же важна, как производительность в рантайме.
