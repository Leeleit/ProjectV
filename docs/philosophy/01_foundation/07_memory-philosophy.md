# Философия аллокаций в нашем движке

Память — это не бесконечное пространство, а самый дефицитный ресурс в высокопроизводительном движке. Студенты привыкли к
`new` и `delete` (или `std::make_unique`). Они думают, что аллокация стоит дёшево. Они не знают про фрагментацию памяти
и стоимость системных вызовов в куче (heap). Если им не запретить стандартные аллокаторы, они превратят твой идеальный
DOD в решето из разрозненных кусков памяти.

---

## Отказ от кучи в рантайме

**Правило №1:** Аллокации общего назначения (`malloc`, `new`, `std::make_unique`) разрешены **только** на этапе
инициализации/загрузки уровня. В игровом цикле — строгий запрет.

Почему?

1. **Фрагментация:** Каждая аллокация в куче оставляет "дыры". Через час работы у тебя 2GB свободной памяти, но нет
   непрерывного блока размером 64KB.
2. **Системный вызов:** `malloc` — это не просто сдвиг указателя. Это обращение к менеджеру памяти ОС, который ищет
   свободный блок, делит его, обновляет таблицы. Это **тысячи** тактов процессора.
3. **Непредсказуемость:** Время аллокации зависит от текущего состояния кучи. Сегодня `new` занял 100 нс, завтра — 10
   мс. В реальном времени это смерть.

> **Метафора:** `malloc` — это заказ постройки нового склада каждый раз, когда тебе нужно положить коробку. И снос
> склада (`free`), когда коробка больше не нужна. Ты платишь архитектору, строителям, сносной бригаде. Через месяц у тебя
> город из полуразрушенных складов, но нет ни одного целого.

## Linear/Arena Allocator — самый быстрый аллокатор

Самый быстрый аллокатор — это просто сдвиг указателя:

```cpp
class LinearAllocator {
    uint8_t* memory;
    size_t capacity;
    size_t offset = 0;

public:
    void* allocate(size_t size) {
        void* ptr = memory + offset;
        offset += size;
        return ptr;
    }

    void reset() { offset = 0; } // Всё "освобождается" за одну операцию
};
```

**Преимущества:**

- O(1) аллокация (просто `ptr += size`)
- Нет фрагментации (память всегда непрерывна)
- Освобождение всех объектов за одну операцию (`reset()`)

**Недостатки:**

- Нельзя освободить отдельный объект
- Подходит для временных данных или данных с одинаковым временем жизни

> **Для понимания:** Linear Allocator — это конвейерная лента. Ты кладёшь детали на ленту одну за другой. Когда все
> детали обработаны, ты не разбираешь каждую с ленты — ты просто сбрасываешь все разом и начинаешь с начала. Идеально для
> обработки кадра: собрал все данные, обработал, сбросил.

## Frame Allocator — память на один кадр

Frame Allocator — это специализированный Linear Allocator для данных, которые живут ровно один кадр:

```cpp
class FrameAllocator {
    static constexpr size_t FRAME_COUNT = 2; // Double buffering
    LinearAllocator frames[FRAME_COUNT];
    size_t current_frame = 0;

public:
    void* allocate(size_t size) {
        return frames[current_frame].allocate(size);
    }

    void next_frame() {
        // Переключаемся на следующий кадр
        current_frame = (current_frame + 1) % FRAME_COUNT;
        // Сбрасываем аллокатор для нового кадра
        frames[current_frame].reset();
    }
};
```

**Что храним в Frame Allocator:**

- Временные массивы трансформаций для GPU
- Строки для отладки (логгирование)
- Результаты промежуточных вычислений
- Команды рендеринга (до отправки в GPU)

> **Метафора:** Frame Allocator — это школьная доска. Ты пишешь на ней весь урок (кадр): формулы, рисунки, заметки.
> После звонка ты не стираешь каждую букву отдельно — ты берёшь тряпку и за одну секунду очищаешь всю доску. Завтрашний
> урок начинается с чистой доски.

## Stack Allocator — для рекурсивных алгоритмов

Иногда нужно временно выделить память внутри функции и освободить при выходе:

```cpp
class StackAllocator {
    uint8_t* memory;
    size_t capacity;
    size_t offset = 0;

public:
    void* allocate(size_t size) {
        void* ptr = memory + offset;
        offset += size;
        return ptr;
    }

    void deallocate_to_marker(size_t marker) {
        offset = marker; // "Откатываем" до маркера
    }

    size_t get_marker() const { return offset; }
};
```

**Использование:**

```cpp
void process_mesh() {
    auto marker = stack_allocator.get_marker();

    // Временные данные для алгоритма
    auto temp_vertices = stack_allocator.allocate(vertex_count * sizeof(Vec3));
    auto temp_indices = stack_allocator.allocate(index_count * sizeof(uint32_t));

    // ... алгоритм мешинга ...

    // Автоматическое освобождение всего, что выделили
    stack_allocator.deallocate_to_marker(marker);
}
```

## Pool Allocator — для однотипных объектов

Если нужно много маленьких объектов одного размера (частицы, события, ноды графа):

```cpp
class PoolAllocator {
    struct Node { Node* next; };
    Node* free_list = nullptr;

public:
    void* allocate() {
        if (free_list) {
            Node* node = free_list;
            free_list = free_list->next;
            return node;
        }
        // ... или выделяем новый блок
    }

    void deallocate(void* ptr) {
        Node* node = static_cast<Node*>(ptr);
        node->next = free_list;
        free_list = node;
    }
};
```

**Преимущества:**

- O(1) аллокация и освобождение
- Нет фрагментации (все объекты одинакового размера)
- Идеально для частиц, событий ECS, пула строк

## Когда что использовать

| Аллокатор            | Когда использовать                          | Время жизни              |
|----------------------|---------------------------------------------|--------------------------|
| **Frame Allocator**  | Временные данные кадра                      | Один кадр                |
| **Stack Allocator**  | Рекурсивные алгоритмы, временные вычисления | До конца функции/области |
| **Pool Allocator**   | Много однотипных объектов (частицы)         | Произвольное             |
| **Linear Allocator** | Загрузка уровня, инициализация              | До перезагрузки уровня   |
| **Heap (malloc)**    | **НИКОГДА в игровом цикле**                 | Только инициализация     |

## Золотые правила

1. **Никакого `malloc` в игровом цикле.** Только на загрузке.
2. **Frame Allocator — твой лучший друг** для временных данных.
3. **Знай время жизни данных:** если данные живут один кадр — Frame, если до конца функции — Stack.
4. **Измеряй:** Tracy покажет, если аллокатор стал узким местом.

> **Метафора итоговая:** Представь, что память — это вода. `malloc` — это бутилированная вода: дорого, пластик, нужно
> идти в магазин. Frame Allocator — это водопровод: открыл кран — вода течёт, закрыл — перестала. Не нужно бутылок, не
> нужно магазинов. В высокопроизводительном движке мы строим водопровод, а не покупаем бутилированную воду.
