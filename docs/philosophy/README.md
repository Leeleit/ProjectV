# Философия архитектуры: когда код диктуется физикой железа

Этот раздел — инструкция по выживанию в условиях, когда у тебя **16 миллисекунд** на миллионы операций. Здесь мы
объясняем, почему наш движок устроен именно так, а не иначе. Не «как» — это в стандартах. А **почему**: какие принципы
лежат в основе архитектурных решений, как мы думаем о производительности, данных и коде.

---

## Глоссарий «на пальцах»

### Cache Line (кэш‑линия)

**64 байта** — это атомарная единица работы с памятью. Представь, что ты берёшь книгу с полки. Ты не можешь взять одну
страницу — ты хватаешь всю книгу целиком. Кэш‑линия — это «захват рукой»: процессор загружает сразу 64 байта, даже если
нужен всего один. Если рядом лежит мусор — он тоже попадёт в кэш. Наша задача — упаковать данные так, чтобы в каждой
«книге» были только нужные страницы.

### Branch Misprediction (ошибка предсказания ветвления)

Процессор — это конвейер. Он не ждёт, пока выполнится текущая инструкция, а начинает следующую. Но когда встречается
`if`, он должен угадать, какая ветка выполнится. Угадал — конвейер летит. Ошибся — вся очередь сбрасывается (штраф 10–20
тактов). Это как ошибка на развилке конвейера: выбрал не ту дорогу — приходится возвращаться и начинать заново.

### DOD (Data‑Oriented Design)

Мышление не объектами, а **данными и их трансформациями**. Вместо «у меня есть объект, какие у него методы?» мы
спрашиваем: «у меня есть массив данных, как мне его быстро обработать?». Код — это инструмент трансформации данных,
ничего больше.

### ECS (Entity Component System)

Композиция вместо наследования. Сущность — это просто ID (ключ). Компоненты — чистые POD‑данные. Системы — функции
трансформации. Никаких алмазных наследований, никаких виртуальных вызовов. Собираем поведение как конструктор LEGO.

### Mesh Shaders

Классический пайплайн (вершинный шейдер → растеризация) мёртв для сложной геометрии. Mesh Shaders позволяют GPU самому
генерировать треугольники на лету, без гигантских Vertex Buffers. Это не хранение архивов распечатанных документов, а *
*генерация голограммы в момент, когда на неё смотрят**.

### SIMD (Single Instruction, Multiple Data)

Одна инструкция обрабатывает 4, 8 или 16 чисел одновременно. Вместо 4 умножений — одно. Авто‑векторизация компилятора
работает в простых случаях, но ломается на сложных циклах. В критических местах мы пишем SIMD вручную — это сложнее, но
предсказуемее.

### Bindless Rendering

«Привяжи один раз, рисуй везде». Вместо того чтобы группировать ресурсы по частоте обновления, мы сваливаем всё в
глобальные массивы (текстуры, матрицы, материалы). Шейдер получает индекс и сам достаёт данные. Это позволяет рисовать
весь мир **одним Draw Call**. Метафора: повар в ресторане получает книгу всех рецептов и ингредиентов сразу, а не бегает
за каждым заказом.

### False Sharing

Размещение атомиков или часто изменяемых данных разных потоков в одной кэш‑линии. Когда один поток пишет в своё поле,
вся кэш‑линия инвалидируется у других ядер. Это неявный mutex. Решение: `alignas(64)` — разнести данные на разные
«столы».

### Pointer Chasing

Беготня по указателям: `entity→component→data→field`. Каждая стрелочка — потенциальный cache miss (100–300 тактов CPU).
Это как поиск документа в архиве по цепочке записок, вместо того чтобы взять стопку сразу.

### Double Buffering

Читаем из State N, пишем в State N+1. Никаких гонок данных, никаких мьютексов. Кадр N читает, кадр N+1 пишется.
Представь две одинаковые книги: один пишет в книгу А, другой читает из книги Б. Потом меняемся.

## Основные принципы

1. **Производительность — не опция, а обязательство.** Медленный код — это архитектурная ошибка, а не характеристика
   железа.
2. **Данные важнее кода.** Структуры данных определяют архитектуру. Сначала проектируем, как данные лежат в памяти,
   потом пишем функции.
3. **Явный контроль.** Никакой скрытой магии в критических путях. Мы знаем стоимость каждой абстракции.
4. **Прагматизм.** Инструменты выбираются под задачу, не под моду. DOD, ECS, Vulkan — потому что они решают конкретные
   проблемы, а не потому что модно.
