# Философия воксельных данных (2026 Edition)

Воксельный движок — это задача управления **потоком данных**. В 2026 году CPU слишком медленный, чтобы трогать каждый
воксель. Наша цель — как можно быстрее перекинуть данные на GPU и заставить видеокарту делать всю тяжелую работу.

---

## Сдвиг Парадигмы: GPU-Driven Rendering

В старых движках (до 2022) пайплайн был таким:

1. CPU: Проход по чанкам, Frustum Culling.
2. CPU: Генерация списка Draw Calls.
3. CPU -> GPU: Заливка Uniforms.
4. GPU: Рисует.

В ProjectV (Modern):

1. CPU: Обновляет изменившиеся данные (Stream in/out).
2. GPU (Compute): Culling (Occlusion & Frustum) миллионов чанков.
3. GPU (Compute/Mesh Shader): Генерация геометрии.
4. GPU: Indirect Draw всего мира за один вызов.

CPU больше не знает, *что* рисуется. Он знает только, *что существует*.

> **Метафора:** Представь фабрику. В старом подходе менеджер (CPU) стоит у каждого станка и говорит: "Теперь делай
> деталь А. Теперь деталь Б. Теперь деталь В". Менеджер бегает, устаёт, тратит время на дорогу.
>
> В новом подходе менеджер один раз загружает чертежи в память станков и говорит: "Работайте". Станки (GPU) сами читают
> чертежи, сами решают, что делать, сами сообщают о готовности. Менеджер свободен для других задач.
>
> **Почему это важно:** CPU — один (или 16-32 ядра). GPU — тысячи ядер. Перенеся работу на GPU, мы распараллеливаем её
> на тысячи потоков. Culling миллиона чанков на CPU — это секунды. На GPU — миллисекунды.

## Воксель — это данные, а не геометрия

### Структура Чанка

Мы не храним `Vertex Array` для каждого чанка в RAM. Это расточительно.
Мы храним **SVO (Sparse Voxel Octree)** или **DAG (Directed Acyclic Graph)**, который компактно описывает объем.

Для рендеринга мы используем **Palette Compression**:

```cpp
struct Chunk {
    uint8_t indices[32*32*32]; // 32KB
    MaterialPalette palette;    // Локальная палитра (напр. 16 материалов)
};
```

Если чанк состоит из одного материала, он схлопывается в 4 байта (Solid Node).

> **Для понимания:** Зачем хранить каждый воксель отдельно, если чанк может быть полностью однородным? SVO позволяет
> представлять огромные однородные области одним узлом. Чанк 32×32×32 из одного материала — это не 32,768 вокселей, а
> одна
> запись: "всё здесь — камень". Это сжатие в тысячи раз. Для типичного ландшафта (много воздуха, много однородных слоёв)
> экономия колоссальная.

## Мешинг: Compute vs Mesh Shaders

### Старый подход: CPU Meshing (Greedy Meshing)

Медленно. Требует синхронизации. Забивает шину PCIe геометрией.

### Новый подход A: Compute Shader Meshing

GPU читает воксельные данные (SSBO) и генерирует вершины в буфер.
Используется для совместимости.

### Новый подход B: Mesh Shaders (Task + Mesh)

Вершина эволюции.

1. **Task Shader:** Решает, виден ли чанк (Cluster Culling).
2. **Mesh Shader:** Генерирует треугольники *на лету* из воксельных данных.

Никаких буферов вершин! Геометрия рождается и умирает в пайплайне GPU. Это экономит гигабайты VRAM.

> **Метафора:** В старом подходе ты печатаешь все документы заранее и носишь их с собой в папке. Тяжело, много бумаги,
> не всё пригодится.
>
> С Mesh Shaders ты генерируешь документ в момент, когда он нужен, и сразу показываешь. Ни папок, ни лишней бумаги.
> Документ "живёт" только в момент показа.
>
> Для вокселей это идеально: не нужно хранить миллионы треугольников в памяти. Мы генерируем их на лету из компактных
> данных вокселей.

## Bindless Resources (Descriptor Indexing)

У нас миллионы чанков. Биндить текстуру на каждый чанк (`vkCmdBindDescriptorSets`) — убийство производительности.
Мы используем **Bindless Rendering**:

```glsl
// Shader
layout(binding = 0) uniform sampler2D all_textures[];

void main() {
    uint textureID = chunk_data.material_id;
    vec4 color = texture(all_textures[textureID], uv);
}
```

Мы биндим **ВСЕ** текстуры игры один раз на старте. Шейдер сам выбирает нужную по индексу.

> **Для понимания:** Это как библиотека, где все книги лежат на полках с номерами. Вместо того чтобы просить
> библиотекаря принести конкретную книгу, ты сам идёшь к полке и берёшь нужный номер. Библиотекарь (CPU) свободен, ты (
> GPU) работаешь сам.

## Что хранить, а что вычислять

**Правило 2026:** Вычисляй всё. Память (Memory Bandwidth) — узкое место. ALU (Compute) — дешевое.

| Данные    | Где хранить              | Формат                                 |
|-----------|--------------------------|----------------------------------------|
| Воксели   | GPU VRAM (Sparse Buffer) | Compressed SVO/DAG                     |
| Нормали   | Нигде                    | Вычисляются в Pixel Shader (dFdx/dFdy) |
| Освещение | GPU VRAM (Voxel Grid)    | Trace on update                        |
| Геометрия | Нигде (Mesh Shaders)     | Генерируется каждый кадр               |

> **Для понимания:** Нормали — классический пример. Зачем хранить 12 байт на вершину, если GPU может вычислить их из
> позиций соседних пикселей за пару инструкций? `dFdx` и `dFdy` — это встроенные функции GLSL, которые дают производные.
> Из них нормаль вычисляется бесплатно. Хранение — это расход памяти и пропускной способности. Вычисление — это почти
> бесплатно на современном GPU.
