# Философия потока данных

Поток данных в игровом движке — это не просто "передача переменных". Это **архитектура границ**, где каждая система
знает только то, что ей нужно, и не знает ничего лишнего. Мы строим поток данных как конвейер на заводе: сырьё входит,
проходит через станки, выходит готовый продукт. Никакого спагетти-кода на стыках.

---

## Границы (Boundaries) — главный принцип

Каждая система в движке имеет чёткие границы входа и выхода. Это как отделы на заводе:

- **Цех физики** получает позиции и скорости, возвращает новые позиции
- **Цех рендеринга** получает позиции и материалы, рисует кадр
- **Цех AI** получает позиции игрока, возвращает решения NPC

Границы определяются **данными**, а не вызовами функций. Системы не вызывают друг друга напрямую. Они кладут результаты
в общие буферы, откуда другие системы их забирают.

> **Метафора:** Представь почтовое отделение. Отправитель (система A) кладёт письмо в ячейку (буфер). Получатель (
> система B) забирает письмо из своей ячейки. Отправитель не знает, кто получатель. Получатель не знает, кто отправитель.
> Они знают только протокол: "письма кладутся сюда, забираются отсюда". Это и есть граница.

## ECS как система границ

В ECS границы реализуются через компоненты:

1. **Система движения** читает `Velocity` и `Position`, пишет новый `Position`
2. **Система коллизий** читает `Position` и `Collider`, пишет `CollisionEvent`
3. **Система физики** читает `CollisionEvent` и `Velocity`, обновляет `Velocity`

Каждая система работает только со своими компонентами. Она не знает, какие ещё системы существуют. Это делает
архитектуру **модульной**: можно добавить новую систему, не переписывая старые.

```cpp
// Плохо: системы вызывают друг друга
physics_system.update();
render_system.update(physics_system.get_positions()); // Спагетти!

// Хорошо: системы работают с общими данными
world.progress(dt); // Все системы работают параллельно с данными
```

> **Для понимания:** В традиционном ООП объекты вызывают методы друг друга: `player->move()`, `player->draw()`. Это
> создаёт тесные связи (coupling). Если изменится `Player` — нужно менять все, кто его вызывает. В ECS системы независимы.
> Они работают с компонентами, а не друг с другом. Изменение одной системы не затрагивает другие, если не меняются
> компоненты на границе.

## Double Buffering: кадр N читает, кадр N+1 пишется

В реальном времени нельзя читать и писать одни и те же данные одновременно. Пока GPU рисует кадр N, CPU готовит кадр
N+1. Это требует **двойной буферизации**.

### Принцип Double Buffering:

- **Буфер A:** данные для текущего кадра (читает GPU)
- **Буфер B:** данные для следующего кадра (пишет CPU)
- После завершения кадра: меняем буферы местами

```cpp
struct FrameData {
    std::array<Transform, MAX_OBJECTS> transforms;
    std::array<Material, MAX_OBJECTS> materials;
};

FrameData frames[2]; // Двойной буфер
int current_frame = 0;

// CPU пишет в следующий кадр
FrameData& write_frame = frames[1 - current_frame];
update_transforms(write_frame.transforms);

// GPU читает текущий кадр
FrameData& read_frame = frames[current_frame];
draw_objects(read_frame.transforms, read_frame.materials);

// В конце кадра: swap
current_frame = 1 - current_frame;
```

> **Метафора:** Представь кухню ресторана. Повар (CPU) готовит блюдо для столика 2 на одной плите. Официант (GPU) несёт
> готовое блюдо для столика 1 с другой плиты. Они не мешают друг другу, потому что используют разные плиты. В конце
> обслуживания они меняются плитами: повар начинает готовить на освободившейся, официант забирает с только что
> приготовленной.

## Передача данных в Vulkan: от ECS к GPU

Граница между CPU и GPU — самая критичная. Здесь нельзя допускать спагетти.

### Шаг 1: Подготовка данных (CPU, ECS)

ECS системы обновляют компоненты. В конце кадра мы собираем **только изменившиеся данные**:

```cpp
// Собираем все трансформы, которые изменились в этом кадре
auto changed_transforms = collect_changed<Transform>();

// Копируем их в GPU-буфер (Staging Buffer)
copy_to_staging_buffer(changed_transforms);
```

### Шаг 2: Загрузка в GPU (Transfer Queue)

Используем отдельную очередь Transfer для асинхронной загрузки:

```cpp
vkCmdCopyBuffer(cmd, staging_buffer, gpu_buffer, ...);
```

### Шаг 3: Шейдерный доступ (Graphics Queue)

Шейдеры читают данные через Bindless Descriptors:

```glsl
// Все трансформы в одном большом буфере
layout(binding = 0) readonly buffer Transforms {
    mat4 transforms[];
};

// Шейдер получает индекс объекта и сам берёт трансформ
mat4 model = transforms[gl_InstanceIndex];
```

> **Для понимания:** Раньше для каждого объекта делали отдельный Uniform Buffer и биндили его перед рисованием. 10,000
> объектов = 10,000 вызовов `vkCmdBindDescriptorSets`. Это убийство производительности. В современном Vulkan мы загружаем
> все данные в один гигантский буфер и используем индексацию. GPU сам берёт нужные данные по индексу. CPU только один раз
> говорит "вот все данные, рисуй".

## Границы синхронизации: Timeline Semaphores

Double Buffering требует синхронизации. Мы используем Timeline Semaphores:

```cpp
// CPU сигнализирует: "я закончил писать кадр N+1"
vkSignalSemaphore(device, semaphore, frame_number + 1);

// GPU ждёт: "не начинай рисовать кадр N+1, пока CPU не закончил"
vkWaitSemaphore(device, semaphore, frame_number + 1);

// GPU сигнализирует: "я закончил рисовать кадр N"
vkSignalSemaphore(device, semaphore, frame_number);

// CPU ждёт: "не начинай писать кадр N+2, пока GPU не закончил с N"
vkWaitSemaphore(device, semaphore, frame_number);
```

Это создаёт конвейер: CPU всегда на один кадр впереди GPU, но они не мешают друг другу.

## Исключения из правил: когда границы размываются

Бывают случаи, когда нужно нарушить чистоту границ:

### Immediate Mode UI (ImGui)

UI рисуется каждый кадр, данные генерируются и сразу отправляются на рендеринг. Здесь Double Buffering излишен — UI
простой и быстрый.

### Debug Visualization

Отладочные линии и боксы генерируются и рисуются в том же кадре. Это okay, потому что это инструмент разработки, не
продакшен-код.

> **Метафора:** На заводе есть главный конвейер (продакшен) и маленькая экспериментальная линия (R&D). На главном
> конвейере строгие правила, двойная буферизация, чёткие границы. На экспериментальной линии можно нарушать правила для
> быстрого прототипирования. Главное — не путать их. Продакшен-код должен быть чистым. Инструменты разработки могут быть
> грязнее.

## Золотые правила потока данных

1. **Одна система — одна ответственность:** не смешивай физику, рендеринг и AI в одной системе
2. **Данные, а не вызовы:** системы общаются через буферы, а не прямые вызовы
3. **Double Buffering везде, где есть CPU-GPU взаимодействие**
4. **Bindless везде, где много объектов:** один буфер, индексация
5. **Синхронизация через Timeline Semaphores:** не через `vkQueueWaitIdle`

## Mermaid диаграмма: Double Buffering и Timeline Semaphores

```mermaid
gantt
    title Double Buffering: Кадр N читает GPU, Кадр N+1 пишет CPU
    dateFormat  HH:mm:ss.SSS
    axisFormat %S.%L s

    section CPU (Пишет данные)
    Кадр N+1 подготовка :crit, cpu1, 2026-01-01 00:00:00.000, 5ms
    Кадр N+1 синхронизация :cpu2, after cpu1, 1ms
    Кадр N+2 подготовка :crit, cpu3, after cpu2, 5ms
    Кадр N+2 синхронизация :cpu4, after cpu3, 1ms

    section GPU (Читает и рисует)
    Кадр N рендеринг :crit, gpu1, 2026-01-01 00:00:01.000, 8ms
    Кадр N синхронизация :gpu2, after gpu1, 1ms
    Кадр N+1 рендеринг :crit, gpu3, after gpu2, 8ms
    Кадр N+1 синхронизация :gpu4, after gpu3, 1ms

    section Синхронизация
    CPU сигнал: кадр N+1 готов :milestone, m1, 2026-01-01 00:00:05.000, 0ms
    GPU ждёт кадр N+1 :milestone, m2, 2026-01-01 00:00:06.000, 0ms
    GPU сигнал: кадр N готов :milestone, m3, 2026-01-01 00:00:09.000, 0ms
    CPU ждёт кадр N :milestone, m4, 2026-01-01 00:00:10.000, 0ms
```

**Объяснение диаграммы:**

- **Верхняя секция (CPU):** Готовит данные для следующего кадра (N+1, N+2)
- **Средняя секция (GPU):** Рендерит текущий кадр (N, N+1)
- **Нижняя секция (Синхронизация):** Timeline Semaphores координируют работу
- **Критические пути (crit):** Подготовка данных и рендеринг идут параллельно
- **Перекрытие:** Пока GPU рисует кадр N, CPU готовит кадр N+1

**Преимущества Double Buffering:**

1. **Нет простоев:** CPU и GPU работают параллельно
2. **Нет гонок:** Данные кадра N читаются только GPU, данные кадра N+1 пишутся только CPU
3. **Предсказуемая задержка:** Всегда 1 кадр между подготовкой и отображением
