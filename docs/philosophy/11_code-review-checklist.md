# Чек‑лист для Code Review (Pull Requests)

Этот чек‑лист — выжимка из философии движка. Используй его при ревью Pull Request'ов, чтобы код соответствовал нашим
принципам производительности, безопасности и читаемости.

---

## Память и производительность

### 1. Структуры данных

- [ ] **Поля отсортированы от большего к меньшему** (`double` → `int` → `short` → `char` → `bool`)
- [ ] **Hot/Cold splitting** применён для разделения часто и редко используемых данных
- [ ] **SoA (Structure of Arrays)** используется для batch processing (физика, рендеринг)
- [ ] **Выравнивание** `alignas(64)` для предотвращения false sharing между потоками
- [ ] **Padding** минимизирован (проверить `sizeof()` и `alignof()`)

### 2. Аллокации

- [ ] **Нет `new`/`malloc` в игровом цикле** (используется `FrameAllocator`, `ArenaAllocator`)
- [ ] **Нет `std::string` в hot‑path** (используется `StringID` или `std::string_view`)
- [ ] **Контейнеры** имеют предварительный резерв (`reserve()`) для избежания реаллокаций
- [ ] **Small Buffer Optimization** используется для маленьких объектов (например, `std::array` вместо `std::vector` для
  фиксированного размера)

### 3. Многопоточность

- [ ] **Нет `std::thread` напрямую** (используется Job System)
- [ ] **Lock‑free структуры** или `std::atomic` для высокочастотных данных
- [ ] **False sharing** предотвращён (`alignas(64)` для разделяемых данных)
- [ ] **Data races** отсутствуют (Tracy или ThreadSanitizer проверены)

---

## Обработка ошибок и безопасность

### 4. Инварианты

- [ ] **Нарушения инвариантов** обёрнуты в `CORE_ASSERT`, а не скрыты за `if (ptr == nullptr) return;`
- [ ] **Все указатели** проверены на `nullptr` перед разыменованием в публичном API
- [ ] **Индексы массивов** проверены на границы в Debug сборке

### 5. Ожидаемые ошибки

- [ ] **Ожидаемые ошибки** возвращаются через `std::expected<T, E>`, а не коды ошибок или исключения
- [ ] **Функции с ошибками** помечены `[[nodiscard]]`
- [ ] **Обработка ошибок** явная (используется `.and_then()`, `.or_else()`, `.transform()`)

### 6. Безопасность типов

- [ ] **Сырые указатели** заменены на `std::span`, `std::reference_wrapper` или умные указатели с явным владением
- [ ] **Целочисленные переполнения** предотвращены (сатурация, проверки)
- [ ] **Неинициализированные переменные** отсутствуют (все переменные инициализированы)

---

## Архитектура и кодстайл

### 7. Data‑Oriented Design

- [ ] **Код думает данными**, а не объектами (массивы вместо отдельных объектов)
- [ ] **Системы** — чистые функции трансформации данных (без скрытого состояния)
- [ ] **ECS** используется для композиции поведения (Flecs)
- [ ] **Virtual методы** отсутствуют в hot‑path (используется compile‑time полиморфизм или `std::variant`)

### 8. Производительность кода

- [ ] **Ветвления (`if`)** минимизированы в горячих циклах (branchless код, сортировка данных)
- [ ] **SIMD/AVX** используется для векторизации критичных вычислений
- [ ] **Inline функции** помечены `inline` или `__forceinline` где необходимо
- [ ] **Кэш‑промахи** минимизированы (линейный доступ к данным, prefetching)

### 9. Читаемость и поддерживаемость

- [ ] **Имена переменных** понятны и отражают семантику (не `a`, `b`, `tmp`)
- [ ] **Комментарии** объясняют «почему», а не «что» (код должен быть самодокументируемым)
- [ ] **Сложные алгоритмы** снабжены ссылками на статьи или объяснения в комментариях
- [ ] **Магические числа** вынесены в константы с понятными именами

---

## Тестирование и отладка

### 10. Тестирование

- [ ] **Инварианты** покрыты unit‑тестами (используется Catch2, Google Test)
- [ ] **Производительность** измерена (Tracy Profiler, benchmarks)
- [ ] **Edge cases** обработаны (пустые массивы, нулевые указатели, крайние значения)
- [ ] **Детерминированность** обеспечена для физики и логики (fixed‑point математика, фиксированный шаг времени)

### 11. Отладка

- [ ] **Логирование** имеет уровни (`CORE_LOG_FATAL`, `CORE_LOG_ERROR`, `CORE_LOG_WARNING`, `CORE_LOG_INFO`)
- [ ] **Debug‑инструменты** включены в сборку (ImGui debug draw, Tracy zones)
- [ ] **Стек‑трейсы** собираются при фатальных ошибках (Crash Dumps)
- [ ] **GPU дебаг** возможен (RenderDoc, Nsight совместимость)

---

## Специфичные для движка

### 12. Графика (Vulkan)

- [ ] **Bindless descriptors** используются для минимизации CPU‑GPU синхронизации
- [ ] **Mesh Shaders** применяются для генерации геометрии на лету
- [ ] **Dynamic Rendering** (Vulkan 1.4) вместо legacy пайплайнов
- [ ] **Memory aliasing** (VMA) для эффективного использования GPU памяти

### 13. Воксели и большие миры

- [ ] **Целочисленные координаты** (`int64_t`) для логики вокселей
- [ ] **Origin Shifting** для рендеринга больших миров без потери точности
- [ ] **SVO сжатие** используется для разреженных воксельных данных
- [ ] **LOD системы** автоматически переключают детализацию

### 14. Время и детерминизм

- [ ] **Fixed Time Step** для физики и логики
- [ ] **Интерполяция** для плавного рендеринга между тиками
- [ ] **Детерминированные вычисления** (fixed‑point математика, одни и те же операции на всех платформах)
- [ ] **Часы** используют монотонный источник времени (`std::chrono::steady_clock`)

---

## Как использовать этот чек‑лист

1. **Перед созданием PR** — пройдись по пунктам самостоятельно.
2. **При ревью** — отмечай нарушения и требуй исправлений.
3. **После мержа** — убедись, что все проверки пройдены в CI/CD.

> **Важно:** Этот чек‑лист — не догма. Бывают обоснованные исключения, но они должны быть задокументированы в
> комментариях к коду.
