# Взаимодействие систем в ECS

В ProjectV системы проектируются как независимые модули обработки данных. Это упрощает расширение функциональности и
обеспечивает возможность параллельного выполнения систем на многоядерных процессорах.

## 1. Синглтоны (Singleton Components)

Глобальные данные, доступные для всех систем (например, `Time`, `Input`, `GameConfig`), хранятся в ECS как
компоненты-синглтоны (компоненты, существующие в единственном экземпляре в рамках мира).

```cpp
// Определение глобальных данных
struct Time {
    float dt;
    float elapsed;
};

// Доступ из любой системы:
const Time* t = world.get<Time>();
float current_dt = t->dt;
```

Преимущество использования синглтонов заключается в отсутствии явных зависимостей между системами, так как они
взаимодействуют через общее состояние мира.

## 2. События и Обсерверы (Events & Observers)

Для выполнения разовых действий при изменении состояния сущностей используются обсерверы (Observers). Это позволяет
избежать проверки условий в каждом кадре для систем с редкими событиями.

### Типы событий:

- `OnAdd`: Срабатывает при добавлении компонента к сущности.
- `OnSet`: Срабатывает при изменении данных существующего компонента.
- `OnRemove`: Срабатывает непосредственно перед удалением компонента.

```cpp
// Реактивная логика через обсервер
world.observer<Health>()
    .event(flecs::OnSet) // Срабатывает при изменении HP
    .each( {
        if (h.current <= 0.0f) {
            it.entity(i).add<DeadTag>(); // Добавление тега смерти
            playDeathSound(it.entity(i)); // Проигрывание звука
        }
    });
```

Использование обсерверов снижает нагрузку на CPU, так как код выполняется только при фактическом изменении данных.

## 3. Выбор метода взаимодействия: Опрос (Polling) против Событий

Для систем с высокой частотой обновления и большим объемом данных предпочтительным является метод опроса (Polling), в то
время как для редких изменений используются события.

### Опрос (Polling):

- Применяется в системах физики, рендеринга и воксельного мешинга.
- Обеспечивает предсказуемую и стабильную нагрузку на кадр.
- Реализуется через стандартный механизм `world.progress()`.

### События (Events):

- Применяются для пользовательского интерфейса (UI), триггеров и редких игровых событий.
- Позволяют снизить среднюю нагрузку, но могут приводить к пиковым всплескам при одновременном срабатывании множества
  событий.

## 4. Управление межсистемными зависимостями

Для передачи результатов вычислений между системами в рамках одного кадра используются промежуточные компоненты (
Intermediate Components).

### Механизм взаимодействия:

1. Система А добавляет промежуточный компонент (например, `CollisionResult`) к сущности.
2. Система Б в этом же кадре считывает данные из `CollisionResult`, выполняет необходимые действия и удаляет компонент.

Данный подход обеспечивает слабую связанность (loose coupling) систем и упрощает их изолированное тестирование. Принципы
организации данных описаны в [Memory Layout](02_memory-layout.md).
