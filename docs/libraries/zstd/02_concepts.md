# Основные понятия Zstd

Архитектура алгоритма, структура данных и принципы работы Zstandard.

## Внутренняя архитектура

Zstd — гибридный алгоритм, объединяющий три техники сжатия:

```
Исходные данные
       ↓
[LZ-компрессор]
  Поиск совпадающих последовательностей (literals + match sequences)
       ↓
[Huffman-кодирование] — для literal bytes
[FSE-кодирование]     — для offset, match length, literal length
       ↓
Zstd Frame (заголовок + сжатые блоки + опциональный checksum)
```

### LZ-компрессия

LZ (Lempel-Ziv) — алгоритм поиска повторяющихся последовательностей. Заменяет повторяющиеся фрагменты ссылками на
предыдущие вхождения:

```
Исходный текст: ABCDEFABCDEFABCDEF
После LZ:       ABCDEF[copy 6 from -6][copy 6 from -12]
```

### FSE (Finite State Entropy)

FSE — алгоритм энтропийного кодирования на основе ANS (Asymmetric Numeral Systems), разработанный Яном Колле.
Преимущества перед Huffman:

- Лучшее сжатие для данных с неравномерным распределением
- Высокая скорость декодирования
- Эффективная работа с небольшими блоками

### Huffman-кодирование

Используется для кодирования literal bytes (байтов, которые не были заменены ссылками LZ). Выбирается автоматически,
когда это выгоднее FSE.

## Структура Zstd Frame

```
┌─────────────────────────────────────────────────────┐
│ Magic Number: 0xFD2FB528 (4 байта)                  │
├─────────────────────────────────────────────────────┤
│ Frame Header:                                       │
│   - Frame Content Size (опционально)                │
│   - Window Size                                     │
│   - Checksum Flag                                   │
│   - Dictionary ID (опционально)                     │
├─────────────────────────────────────────────────────┤
│ Block 1: [Block Header] [Compressed/Raw Data]       │
│ Block 2: ...                                        │
│ ...                                                 │
├─────────────────────────────────────────────────────┤
│ Checksum (опционально, xxHash64)                    │
└─────────────────────────────────────────────────────┘
```

### Magic Number

Каждый Zstd frame начинается с magic number `0xFD2FB528`. Это позволяет автоматически определить формат данных.

### Frame Header

Содержит метаданные для декодирования:

- **Frame Content Size** — размер исходных данных (если известен)
- **Window Size** — размер буфера для LZ-декодирования
- **Checksum Flag** — наличие xxHash64 контрольной суммы
- **Dictionary ID** — идентификатор словаря (если используется)

### Block Types

Каждый блок имеет тип:

| Тип            | Описание                                   |
|----------------|--------------------------------------------|
| **Compressed** | Сжатый блок (LZ + FSE/Huffman)             |
| **Raw**        | Несжатые данные (если сжатие невыгодно)    |
| **RLE**        | Run-Length Encoding (все байты одинаковые) |

Максимальный размер блока — `ZSTD_BLOCKSIZE_MAX` (128 KB по умолчанию).

## Уровни сжатия и производительность

| Уровень | Скорость сжатия | Скорость распаковки | Соотношение  | Сценарий        |
|---------|-----------------|---------------------|--------------|-----------------|
| -5      | ~1500 MB/s      | ~1500 MB/s          | Низкое       | Сетевые буферы  |
| 1       | ~600 MB/s       | ~1200 MB/s          | Хорошее      | Частые операции |
| 3       | ~350 MB/s       | ~1100 MB/s          | Хорошее      | По умолчанию    |
| 10      | ~60 MB/s        | ~1100 MB/s          | Высокое      | Файловый архив  |
| 19      | ~5 MB/s         | ~1100 MB/s          | Максимальное | Дистрибутив     |

Важно: скорость распаковки не зависит от уровня сжатия. Это позволяет использовать максимальное сжатие для данных,
которые будут читаться многократно.

## Контексты: CCtx и DCtx

### CCtx (Compression Context)

Контекст компрессора содержит:

- Таблицы LZ-поиска
- Рабочие буферы
- Параметры сжатия

Создание контекста — дорогая операция. Переиспользуйте контексты:

```cpp
ZSTD_CCtx* cctx = ZSTD_createCCtx();

// Многократное использование
for (auto& data : dataSet) {
    ZSTD_compressCCtx(cctx, ...);
    ZSTD_CCtx_reset(cctx, ZSTD_reset_session_only);
}

ZSTD_freeCCtx(cctx);
```

### DCtx (Decompression Context)

Контекст декомпрессора. Также рекомендуется переиспользовать:

```cpp
ZSTD_DCtx* dctx = ZSTD_createDCtx();

// Многократное использование
for (auto& compressed : compressedSet) {
    ZSTD_decompressDCtx(dctx, ...);
}

ZSTD_freeDCtx(dctx);
```

## Мультифреймовая структура

Zstd поддерживает конкатенацию нескольких фреймов в один поток:

```
[Frame 1][Frame 2][Frame 3]
```

Декомпрессор автоматически обработает все фреймы последовательно. Это полезно для:

- Инкрементального сжатия (добавление данных к существующему архиву)
- Параллельного сжатия разных частей данных
- Разделения данных на независимые фрагменты

## Window Size

Window Size определяет, насколько далеко компрессор может искать совпадения:

- Большой Window Size → лучшее сжатие → больше памяти
- Малый Window Size → хуже сжатие → меньше памяти

По умолчанию Window Size определяется автоматически на основе размера данных. Можно ограничить:

```cpp
ZSTD_CCtx_setParameter(cctx, ZSTD_c_windowLog, 17);  // 2^17 = 128 KB
```

При декомпрессии декодор должен поддерживать Window Size, использованный при сжатии.

## Checksum

Zstd поддерживает опциональную контрольную сумму xxHash64:

```cpp
ZSTD_CCtx_setParameter(cctx, ZSTD_c_checksumFlag, 1);
```

Checksum добавляет 4 байта в конец фрейма и позволяет обнаружить повреждение данных.

## Content Size

Размер исходных данных может быть записан в заголовок фрейма:

```cpp
ZSTD_CCtx_setParameter(cctx, ZSTD_c_contentSizeFlag, 1);
```

Это позволяет:

- Заранее выделить буфер нужного размера
- Проверить целостность данных
- Ускорить декомпрессию (не нужно угадывать размер)

## Связь между параметрами

```
Compression Level
       ↓
Определяет:
  - Window Size
  - Hash Table Size
  - Chain Log
  - Search Depth
  - Strategy
       ↓
Влияет на:
  - Скорость сжатия
  - Степень сжатия
  - Использование памяти
```

Более высокий уровень сжатия увеличивает все параметры поиска, что улучшает сжатие, но замедляет процесс.
