# Философия Vulkan в ProjectV (Modern)

Vulkan — это не просто API, это доступ к "голому железу". В 2026 году Vulkan сильно изменился. Мы используем **Modern
Vulkan (1.3+)**, отбрасывая легаси-груз.

---

## Dynamic Rendering (Vulkan 1.3)

Мы **не используем** `VkRenderPass` и `VkFramebuffer`. Это архаизм, созданный для мобильных GPU 2015 года.
Мы используем `VK_KHR_dynamic_rendering`.

* Нет прекомпиляции RenderPass объектов.
* Начало рендеринга — просто вызов `vkCmdBeginRendering`.
* Проще код, гибче пайплайн.

---

## Bindless Descriptors (Descriptor Indexing)

"Bind once, draw everywhere."
Вместо того чтобы группировать ресурсы по частоте обновления (Set 0: Global, Set 1: Material, Set 2: Object), мы
сваливаем всё в глобальные массивы (Unbounded Arrays).

* Массив всех текстур.
* Массив всех матриц объектов (SSBO).
* Массив всех материалов.

Шейдер получает индекс (через `gl_InstanceIndex` или Push Constant) и сам достает данные.
Это позволяет рисовать весь мир **одним Draw Call** (MultiDrawIndirect).

> **Метафора:** Представь ресторан. Каждый Draw Call — это поход официанта на кухню с заказом. "Шеф, сделай стейк для
> столика 5!" — официант бежит. "Теперь салат для столика 3!" — снова бежит. 10,000 объектов = 10,000 забегов официанта.
> Дверь на кухню (шина PCI-E) забита, официанты толкаются.
>
> **Bindless:** Официант один раз приносит на кухню книгу со всеми заказами дня. "Шеф, вот все заказы. Они
> проиндексированы: страница 1 — стейк, страница 2 — салат. Ты сам решай, что готовить, вот индексы". Один поход — и
> шина
> свободна. Шеф (GPU) сам берёт нужные данные из книги по индексу, не дёргая официанта (CPU) каждый раз.
>
> **Почему это важно:** CPU-GPU синхронизация — это bottleneck. Каждый `vkCmdBindDescriptorSets` — это синхронизация,
> передача данных через медленную шину PCI-E. Bindless убирает эти передачи. CPU один раз загружает все данные в VRAM,
> потом просто говорит "рисуй с индексом N". GPU делает всю работу самостоятельно.

## Mesh Shaders (VK_EXT_mesh_shader)

Классический пайплайн (Input Assembler -> Vertex Shader) мертв для сложной геометрии.
Мы используем **Task & Mesh Shaders**:

1. **Task Shader:** Группирует работу, делает culling целых кластеров (meshlets).
2. **Mesh Shader:** Генерирует вершины и индексы для rasterizer'а.

Это позволяет реализовать **GPU-Driven LOD** и процедурную геометрию (воксели!) эффективнее, чем когда-либо.

> **Для понимания:** В классическом пайплайне CPU отправляет на GPU уже готовые вершины. Миллионы вершин. Шина забита.
> Mesh Shaders позволяют GPU самому генерировать геометрию. CPU отправляет только описание: "нарисуй чанк с
> координатами (
> X, Y, Z)". GPU читает данные вокселей из VRAM и сам решает, какие треугольники рисовать. Это переносит работу с CPU на
> GPU, освобождая шину и CPU для других задач.

## Synchronization 2.0 (Synchronization2)

Мы используем `vkCmdPipelineBarrier2`.

* Более понятный API.
* Точное указание Stage и Access масок.
* Меньше шансов выстрелить себе в ногу (хотя всё еще можно).

---

## Память: VMA и ReBAR

* **Vulkan Memory Allocator (VMA):** Стандарт де-факто. Мы не пишем свой аллокатор.
* **Resizable BAR (ReBAR):** Мы оптимизируем передачу данных, предполагая наличие ReBAR (прямой доступ CPU ко всей
  VRAM). Но имеем fallback через Staging Buffers.

---

## Validation Layers & GPU Assisted Validation

В Debug мы включаем не просто validation layers, а **GPU Assisted Validation** (
`VK_VALIDATION_FEATURE_ENABLE_GPU_ASSISTED_EXT`).
Это позволяет ловить:

* Out of bounds чтение/запись в шейдерах (Descriptor Indexing это любит).
* Бесконечные циклы в шейдерах.

---

## Почему Vulkan, а не OpenGL/DirectX11?

> **Для понимания:** OpenGL и DirectX11 — это "автоматические коробки передач". Ты говоришь "нарисуй треугольник",
> драйвер решает, как это сделать. Удобно, но ты не контролируешь процесс. Если драйвер сделает неоптимально — ты не
> можешь это исправить.
>
> Vulkan — это "механическая коробка". Ты сам управляешь памятью, синхронизацией, пайплайном. Сложнее? Да. Но ты
> получаешь полный контроль. Для воксельного движка, где нужно обрабатывать миллионы объектов, этот контроль критичен.
> Ты
> можешь оптимизировать каждый аспект, вместо того чтобы надеяться на драйвер.
