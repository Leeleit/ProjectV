# Абстракции с Нулевой Стоимостью (Zero-Cost Abstractions)

В ProjectV мы любим C++. Не за сложность, а за **контроль**. "Zero-Cost Abstraction" означает, что **красивый код
компилируется в такой же быстрый ассемблер, как и написанный вручную**.

Это фундаментальный принцип современного C++: мы отказались от выбора между читаемостью и производительностью.
В 2026 году стандарт C++ — это мощнейшее оружие, если уметь им пользоваться.

---

## Что такое Zero-Cost?

Stroustrup сформулировал два принципа:

1. **You don't pay for what you don't use.** — Ты не платишь за то, что не используешь.
2. **What you do use, you couldn't hand-code any better.** — То, что используешь, нельзя написать лучше вручную.

### Пример: `std::unique_ptr`

```cpp
// Красиво и безопасно:
{
    std::unique_ptr<int> p = std::make_unique<int>(42);
    // ... p автоматически удалится ...
}

// Вручную (Си-стайл):
{
    int* p = new int(42);
    // ... не забыть delete p ...
    delete p;
}
```

**Результат:** Ассемблерный код идентичен! `unique_ptr` — это Zero-Cost. Он даёт безопасность типов и RAII бесплатно.

### Анти-Пример: `std::shared_ptr`

```cpp
std::shared_ptr<int> p = std::make_shared<int>(42);
```

**Результат:** Атомарный счетчик ссылок (медленно!), Control Block в куче (фрагментация!).
Это **НЕ** Zero-Cost. Мы используем его только там, где действительно нужно разделяемое владение (редко).

> **Для понимания:** Почему атомарный счётчик — это медленно? `shared_ptr` использует атомарные операции (`fetch_add`,
`fetch_sub`) для подсчёта ссылок. Атомарные операции требуют синхронизации между ядрами CPU и обходят кэш. На одном ядре
> это "дёрганье" mutex-like механизма. На многих ядрах — это настоящая bottleneck-точка, где все потоки толкаются. Для
> UI,
> где у тебя 10-20 объектов — незаметно. Для 400,000 вокселей — смерть.

## Философия выбора C++

### Почему C++ для воксельного движка?

Воксельный движок работает с миллионами элементов в реальном времени. Это требует:

1. **Контроля над памятью** — мы решаем, где и как лежат данные
2. **Отсутствия garbage collection** — предсказуемая latency
3. **Компиляторных оптимизаций** — template metaprogramming, inlining
4. **Прямого доступа к железу** — SIMD, intrinsics, memory-mapped I/O

C++ — единственный язык, который даёт всё это без компромиссов в производительности.

### Эволюция C++: от C с классами до C++26

| Эпоха        | Стандарт | Ключевые фичи                          | Философия                            |
|--------------|----------|----------------------------------------|--------------------------------------|
| Классический | C++98    | Классы, шаблоны, STL                   | "C с классами"                       |
| Современный  | C++11/14 | Lambda, move semantics, smart pointers | "Не плати за то, что не используешь" |
| Продвинутый  | C++17/20 | Concepts, Ranges, Modules              | "Выразительность без overhead"       |
| Будущее      | C++23/26 | Reflection, Pattern matching           | "Метапрограммирование для всех"      |

Мы живём в эпоху, когда C++ даёт абстракции высокого уровня без потери производительности.

## Zero-Cost инструменты ProjectV

### 1. Templates (Шаблоны)

Шаблоны C++ — это генератор кода на этапе компиляции.

```cpp
template <typename T>
T add(T a, T b) { return a + b; }

// Компилятор генерирует:
int add_int(int a, int b) { return a + b; }
float add_float(float a, float b) { return a + b; }
glm::vec3 add_vec3(glm::vec3 a, glm::vec3 b) { return a + b; }
```

Никаких виртуальных вызовов! (В отличие от Java/C# дженериков). Каждый тип получает свою оптимизированную функцию.

> **Для понимания:** В Java/C# дженерики — это "стирание типов". В рантайме `List<Integer>` и `List<String>` — это одно
> и то же: `List<Object>`. Компилятор вставляет касты. В C++ шаблоны — это генератор кода. `std::vector<int>` и
`std::vector<float>` — это два совершенно разных класса, скомпилированных отдельно. Каждый оптимизирован под свой тип.
> Нет кастов, нет проверок в рантайме.

### 2. `constexpr` (Вычисления при компиляции)

```cpp
constexpr int fib(int n) {
    return (n <= 1) ? n : fib(n - 1) + fib(n - 2);
}

constexpr int val = fib(20); // Вычисляется на этапе компиляции
```

Результат `val` будет вычислен *во время компиляции*. В бинарнике будет просто константа `6765`.
Это **Negative-Cost Abstraction** — быстрее, чем вычисление в рантайме!

**Применение в ProjectV:**

- Таблицы синусов для генерации ландшафта
- Константы движка (размеры чанков, magic numbers)
- Compile-time валидация конфигов

### 3. Move Semantics (Семантика перемещения)

```cpp
std::vector<BigObject> createObjects() {
    std::vector<BigObject> result;
    result.push_back(BigObject{...}); // Move, не copy!
    return result; // NRVO или move
}

auto objects = createObjects(); // Возврат по значению — бесплатно
```

Мы не копируем данные. Мы "крадём" указатель. Это делает возврат тяжёлых объектов из функций бесплатным.

> **Для понимания:** Представь, что ты переезжаешь. Копирование — это когда ты покупаешь такой же диван на новой
> квартире и выбрасываешь старый. Move — это когда ты просто забираешь диван с собой. Первый вариант требует денег и
> времени, второй — только обновления адреса в голове. Move semantics — это "кража" ресурсов у временного объекта,
> который
> всё равно умрёт.

### 4. Inline Expansion

Компилятор может встроить (inline) маленькую функцию прямо в место вызова.

```cpp
// Исходный код:
inline float square(float x) { return x * x; }

float result = square(3.0f) + square(4.0f);

// После компиляции (примерно):
float result = (3.0f * 3.0f) + (4.0f * 4.0f);
```

Нет вызова функции. Нет прыжка. Нет накладных расходов. Просто математика.

## Современные Zero-Cost абстракции (C++20/23/26)

### Concepts (C++20): Красивые ограничения типов

Вместо SFINAE и нечитаемых ошибок шаблонов мы используем **Concepts**. Это позволяет компилятору выбирать наиболее
эффективную реализацию и давать внятные ошибки.

```cpp
// Плохо (C++17): SFINAE ад
template <typename T, typename = std::enable_if_t<std::is_integral_v<T>>>
T gcd(T a, T b) { ... }

// Хорошо (C++20): Читаемо и мощно
template <std::integral T>
T gcd(T a, T b) { ... }

// Ещё лучше (Terse Syntax):
auto gcd(std::integral auto a, std::integral auto b) { ... }
```

**Zero-Cost:** Concepts — это compile-time проверки. В рантайме их нет.

Мы используем концепты для **статического полиморфизма**:

```cpp
// Вместо виртуальных функций:
template <Renderable T>
void render(T& object) {
    object.draw(); // Проверка наличия draw() на этапе компиляции
}

// Компилятор генерирует специализированную функцию для каждого типа
```

### Modules (C++20): Смерть #include

Мы минимизируем использование `#include` в пользу `import`.

**Проблема с #include:**

```cpp
#include <vector>    // ~50,000 строк кода парсится в каждом TU
#include <string>    // ~20,000 строк
#include <algorithm> // ~30,000 строк
// 100+ файлов = миллионы строк компилируются многократно
```

**Решение — Modules:**

```cpp
import std;           // Компилируется один раз
import ProjectV.Core; // Наш модуль
```

*Заметка: Взаимодействие с легаси-библиотеками (Vulkan, SDL) всё ещё требует хедеров, но наш код должен быть модульным.*

**Zero-Cost:** Модули ускоряют компиляцию, не влияя на рантайм.

### Deducing This (C++23): Убийца CRTP

CRTP (Curiously Recurring Template Pattern) — это хак для статического полиморфизма. C++23 делает его элегантным.

```cpp
// C++17 CRTP (Old School) — сложно и error-prone
template <typename Derived>
struct Base {
    void update() { static_cast<Derived*>(this)->impl_update(); }
};

// C++23 Explicit Object Parameter — чисто и безопасно
struct Base {
    template <typename Self>
    void update(this Self&& self) {
        self.impl_update(); // Чисто, без cast
    }
};
```

**Zero-Cost:** Нет виртуальных вызовов, нет runtime overhead.

### `std::bit_cast` (C++20): Легальный Type Punning

Вместо `reinterpret_cast` и UB с `union`, мы используем легальные инструменты для работы с битами.

```cpp
float f = 1.0f;
uint32_t u = std::bit_cast<uint32_t>(f); // Легально и бесплатно

// Было бы UB:
// uint32_t u = *reinterpret_cast<uint32_t*>(&f);
```

**Zero-Cost:** Это `memcpy`, который понимает `constexpr`. Компилятор оптимизирует его в ничто.

### `std::start_lifetime_as` (C++23): Работа с сырой памятью

Для работы с буферами сети, файлами, memory-mapped данными. Позволяет трактовать массив байт как массив объектов без
аллокации и без UB.

```cpp
// Легально начинаем lifetime объектов в сырой памяти
auto* voxels = std::start_lifetime_as<Voxel>(buffer, count);
```

### Static Reflection (C++26): Конец бойлерплейта

Мы используем статическую рефлексию (P2996) вместо макросов и кодогенераторов для сериализации и UI.

```cpp
// Примерный синтаксис (зависит от финального стандарта)
template <typename T>
void serialize(const T& obj) {
    template for (constexpr auto member : ^T) {
        write(member.name, obj.[:member:]);
    }
}
```

Это позволяет писать универсальный код сериализации, инспекторы в редакторе и сетевой код без бойлерплейта.

**Zero-Cost:** Рефлексия происходит на этапе компиляции. В рантайме — только данные.

## Абстракции, которые стоят (Non-Zero-Cost)

Не всё в C++ бесплатно. Мы выбираем эти абстракции осознанно.

### `virtual` — косвенный вызов

```cpp
struct Base {
    virtual void update() = 0; // vtable lookup
};

// Стоимость: 2-10 тактов на косвенный вызов + возможный cache miss
```

> **Метафора:** Представь библиотеку. Ты хочешь вызвать метод объекта. Без `virtual` ты идёшь к нужной полке — ты знаешь
> адрес функции, она лежит там же, где и другие функции этого класса. С `virtual` ты сначала идёшь к каталогу (vtable),
> ищешь в нём адрес, потом идёшь по этому адресу. Адрес может быть anywhere — в другой секции памяти, на другой
> кэш-линии.
> Каждый `virtual` вызов — это дополнительный прыжок через каталог. Если прыжок уходит в некэшированную память —
> процессор
> ждёт 100+ тактов.

**Когда использовать:** Когда количество объектов мало и архитектура требует полиморфизма.

**Альтернативы:** `std::variant`, шаблоны, ECS.

### `std::shared_ptr` — атомарный счётчик

```cpp
auto ptr = std::make_shared<int>(42); // atomic refcount
```

**Стоимость:** Атомарные операции на каждый copy/move. Control block в куче.

**Когда использовать:** Разделяемое владение между системами (редко).

**Альтернативы:** `unique_ptr` + сырые указатели, ECS ownership.

### `std::function` — type erasure

```cpp
std::function<void(int)> callback = [](int x) { ... };
```

**Стоимость:** Аллокация, виртуальный вызов, невозможность inline.

**Когда использовать:** Callback API, плагины.

**Альтернативы:** Шаблоны, `std::function_ref` (C++26), прямые указатели на функции.

### Exceptions — стек размотки

```cpp
throw std::runtime_error("oops");
```

**Стоимость:** Zero-cost happy path, но heavy throw/catch. Увеличивает размер бинарника.

**Когда использовать:** Исключительные ситуации (не в hot path).

**Альтернативы:** `std::expected`, `Result<T, E>` паттерн.

## Правила ProjectV

### 1. Доверяй компилятору

Современные компиляторы (Clang/GCC/MSVC) умные. Они могут:

- Заинлайнить шаблонный код в ноль инструкций
- Векторизовать циклы (SIMD auto-vectorization)
- Удалить мёртвый код (DCE)
- Вычислить constexpr выражения

Не пиши "оптимизированный" код, который мешает компилятору.

### 2. Избегай `virtual` в hot path

Виртуальные функции ломают инлайнинг и кэш. Используй шаблоны (CRTP/Deducing This) или `std::variant`.

### 3. Используй сильные типы

Создавай сильные типы. Это бесплатно, но защищает от ошибок.

```cpp
// Плохо: magic types
void setVoxel(int x, int y, int z, int type);

// Хорошо: сильные типы
struct VoxelCoord { int x, y, z; };
struct VoxelType { uint8_t value; };

void setVoxel(VoxelCoord coord, VoxelType type);
```

**Zero-Cost:** После компиляции — те же `int` и `uint8_t`.

### 4. Измеряй, не гадай

Используй Compiler Explorer (Godbolt) чтобы видеть ассемблер. Используй Tracy чтобы видеть производительность.
