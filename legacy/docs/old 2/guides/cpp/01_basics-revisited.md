## 1. Const Correctness (Контракт на чтение)

Используйте `const` везде, где это возможно. Это не просто "подсказка", а жесткий контракт, который позволяет
компилятору оптимизировать код и защищает вас от случайного изменения данных.

```cpp
// Константный параметр: функция обещает не менять меш
void renderMesh(const Mesh& mesh) {
    // mesh.vertices.clear(); // Ошибка компиляции!
}

class Camera {
public:
    // Константный метод: не меняет состояние камеры
    glm::mat4 getProjectionMatrix() const {
        return m_projection;
    }
private:
    glm::mat4 m_projection;
};
```

**Для ProjectV:** Все компоненты ECS (Transform, Velocity, Health) должны быть `const` в системах, которые только читают
их. Это позволяет flecs оптимизировать доступ к данным.

## 2. References vs Pointers (Ссылки против Указателей)

**Золотое правило:** Используйте ссылки по умолчанию, указатели — только когда это необходимо.

* **Ссылки (`&`)**: "Безопасный псевдоним". Не могут быть `null` (в нормальных условиях), не могут быть переприсвоены
  другому объекту. Идеальны для параметров функций.
* **Указатели (`*`)**: Нужны для:
  1. Опциональности (может быть `nullptr`).
  2. Переприсваивания (указатель может начать указывать на другой объект).
  3. Взаимодействия с C-API (Vulkan, SDL).

```cpp
// Передача по ссылке (быстро, без копирования, безопасно)
void updateTransform(Transform& transform);

// Передача по указателю (если трансформ может отсутствовать)
void applyPhysics(Transform* transform) {
    if (transform) {
        // ...
    }
}
```

**Для ProjectV:** В ECS системах flecs использует ссылки для компонентов. Указатели нужны только для опциональных
компонентов или взаимодействия с C-библиотеками.

## 3. Auto (Вывод типов)

`auto` делает код чище, но его нужно использовать с умом.

* **Хорошо**: Для итераторов, сложных шаблонных типов или когда тип очевиден из правой части.
* **Плохо**: Когда тип неясен и важен для понимания логики.

```cpp
// Хорошо: тип очевиден
auto position = glm::vec3(0.0f);
auto it = entityMap.find(id);

// ВАЖНО: auto по умолчанию копирует!
auto mesh = getLargeMesh(); // КОПИРОВАНИЕ! Очень медленно.
const auto& meshRef = getLargeMesh(); // Ссылка. Быстро.
```

**Для ProjectV:** Используйте `auto` для итераторов flecs и сложных типов Vulkan. Всегда проверяйте, не создаете ли вы
случайную копию больших данных.

## 4. Scope & Lifetime (Время жизни)

Понимать, где "живет" ваш объект — критически важно для предотвращения крашей.

> **Метафора:** Представь рабочий стол. **Стек** — это документы, лежащие прямо перед тобой. Ты берёшь документ,
> работаешь с ним, кладёшь обратно. Когда заканчиваешь задачу — убираешь все документы со стола одним движением. Это
> быстро, автоматично, но места на столе мало (обычно 1-8 MB).
>
> **Куча** — это архив в соседнем помещении. Когда тебе нужен документ из архива, ты идёшь туда, ищешь нужную полку,
> берёшь документ, возвращаешься. Это медленно. Но архив огромен — гигабайты памяти. Когда закончил — нужно не забыть
> вернуть документ на место, иначе архив переполнится (memory leak).

* **Stack (Стек)**: Автоматические переменные. Быстро, память освобождается сама при выходе из `{ scope }`. Используйте
  для 90% задач. Ограничен по размеру (обычно 1-8 MB).
* **Heap (Куча)**: Динамическая память (`new`/`malloc`). Медленно, требует ручного (или через умные указатели)
  управления. Используйте для больших данных (текстуры, буферы), размер которых неизвестен заранее.

```cpp
void createVoxel() {
    Voxel v; // Живет на стеке, умрет в конце функции
} // <--- v удален здесь

Voxel* vPtr = nullptr;
{
    Voxel stackVoxel;
    vPtr = &stackVoxel;
}
// vPtr теперь "висячий" (dangling pointer)! Использование приведет к UB/крашу.
```

> **Для понимания:** Почему стек быстрый? Процессор имеет специальный регистр — Stack Pointer (RSP на x86-64). Выделение
> памяти на стеке — это одна инструкция: `sub rsp, 32`. Освобождение — `add rsp, 32`. Никакого поиска свободного места,
> никакого фрагментации. Просто двигаешь указатель.
>
> Куча — это сложная система. `malloc` должен найти свободный блок подходящего размера, возможно разделить большой блок
> на части, учесть выравнивание. `free` должна пометить блок как свободный, возможно объединить с соседними свободными
> блоками. Это сотни инструкций вместо одной.

**Для ProjectV:** Vulkan ресурсы (буферы, текстуры) живут в куче и управляются через RAII. Игровые объекты в ECS живут в
специально выделенных массивах flecs.

## 5. Примеры для ProjectV

### Const correctness в ECS

```cpp
// Система рендеринга только читает Transform и Mesh
void renderSystem(flecs::iter& it, const Transform* transforms,
                  const Mesh* meshes) {
    for (auto i : it) {
        // transforms[i] и meshes[i] - const, нельзя изменить
        renderMesh(transforms[i], meshes[i]);
    }
}

// Система физики изменяет Transform
void physicsSystem(flecs::iter& it, Transform* transforms,
                   const Velocity* velocities) {
    for (auto i : it) {
        // transforms[i] можно изменять, velocities[i] - только читать
        transforms[i].position += velocities[i].value * it.delta_time();
    }
}
```

### References vs pointers в Vulkan

```cpp
// Хорошо: ссылка для обязательного параметра
void bindBuffer(VkCommandBuffer& cmdBuffer, VkBuffer buffer) {
    vkCmdBindVertexBuffers(cmdBuffer, 0, 1, &buffer, &offset);
}

// Хорошо: указатель для опционального параметра
void setDebugName(VkDevice device, VkBuffer* buffer, const char* name) {
    if (buffer && enableDebugNames) {
        VkDebugUtilsObjectNameInfoEXT nameInfo{};
        nameInfo.sType = VK_STRUCTURE_TYPE_DEBUG_UTILS_OBJECT_NAME_INFO_EXT;
        nameInfo.objectType = VK_OBJECT_TYPE_BUFFER;
        nameInfo.objectHandle = (uint64_t)*buffer;
        nameInfo.pObjectName = name;
        vkSetDebugUtilsObjectNameEXT(device, &nameInfo);
    }
}
```

## 6. Распространенные ошибки и решения

### Ошибка: Висячие ссылки/указатели

**Проблема:** Ссылка/указатель на объект, который уже уничтожен.
**Решение:** Всегда следите за временем жизни объектов. Используйте умные указатели для ресурсов с динамическим временем
жизни.

### Ошибка: Неправильное использование const

**Проблема:** Отсутствие `const` там, где он нужен.
**Решение:** Добавляйте `const` ко всем параметрам, которые функция не изменяет. Это защита от ошибок и подсказка
компилятору для оптимизаций.

### Ошибка: Копирование больших объектов

**Проблема:** `auto` создает копию вместо ссылки.
**Решение:** Используйте `const auto&` или `auto&` для больших объектов.

### Ошибка: Утечка памяти Vulkan объектов

**Проблема:** Vulkan объекты не освобождаются.
**Решение:** Всегда используйте RAII обертки для Vulkan объектов.
