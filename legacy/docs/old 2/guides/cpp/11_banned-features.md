# Запрещенные возможности C++

В ProjectV некоторые стандартные возможности C++ запрещены из-за их негативного влияния на производительность (Runtime
Overhead), время компиляции или предсказуемость (Real-Time).

> **Связь с философией:** Этот документ — прямое следствие [08_anti-patterns.md](../../philosophy/08_anti-patterns.md).
> Запрещённые фичи — это не "стриктность ради стриктности", а прагматичные решения для real-time системы.

## 1. Exceptions (Исключения)

**Статус:** ⛔ ЗАПРЕЩЕНО в коде движка.

### Почему?

* **Binary Size:** Таблицы размотки стека (unwind tables) увеличивают бинарник на 10-20%.
* **Runtime Overhead:** Даже если исключение не брошено, наличие `try/catch` может мешать оптимизатору.
* **Непредсказуемость:** Бросок исключения — это тяжелая операция (аллокации, проход по стеку), недопустимая в Real-Time
  цикле.

> **Аппаратный контекст:** При броске исключения процессор должен:
> 1. Найти обработчик в таблице исключений (проход по памяти)
> 2. Размотать стек — вызвать деструкторы всех локальных объектов
> 3. Найти подходящий catch-блок
>
> Это может занять микросекунды — вечность для игрового движка, работающего в рамках 16.67ms бюджета кадра. Исключения
> предназначены для "исключительных" ситуаций, а не для управления потоком.

### Альтернатива

Используйте коды ошибок, `std::optional`, или `std::expected` (C++23).
Для фатальных ошибок (нарушение инвариантов) — `ASSERT` и падение (`std::terminate`).

```cpp
// Плохо:
try {
    init_vulkan();
} catch (const std::exception& e) { ... }

// Хорошо:
if (!init_vulkan()) {
    log_error("Failed to init Vulkan");
    return false;
}
```

## 2. RTTI (Run-Time Type Information)

**Статус:** ⛔ ЗАПРЕЩЕНО. Компилируем с `-fno-rtti`.

### Почему?

* `dynamic_cast` — это медленный проход по дереву наследования (строки, сравнения).
* `typeid` — лишние данные в vtable.

> **Аппаратный контекст:** `dynamic_cast<Base*>(derived)` выполняет:
> 1. Поиск типа в иерархии наследования (проход по связному списку)
> 2. Сравнение строк имён типов (strlen + strcmp)
> 3. Возможно, несколько попыток при множественном наследовании
>
> Это O(n) от глубины иерархии и включает доступ к памяти, разбросанной по куче. В ECS тип компонента известен
> статически — никаких runtime проверок не нужно.

### Альтернатива

* **ECS:** Компоненты — это данные, тип определяется ID компонента.
* **Tag Dispatching:** `enum Type { TYPE_A, TYPE_B }`.
* **Static Polymorphism:** `std::variant` + `std::visit`.

## 3. STL в Hot Path

**Статус:** ⚠️ ОГРАНИЧЕНО.

### Почему?

* `std::vector`: В Debug-режиме (MSVC) очень медленный из-за проверок итераторов.
* `std::shared_ptr`: Атомарный инкремент/декремент счетчика — убийца производительности в многопоточной среде.
* `std::map`: Каждый узел — отдельная аллокация -> Cache Miss.

### Альтернатива

* Используйте `eastl` или собственные контейнеры (`PodVector`, `FlatMap`).
* Для владения используйте `std::unique_ptr` или сырые указатели (если владение внешнее).
* Вместо `std::function` (аллокация!) используйте лямбды или функциональные указатели.

## 4. `volatile` для многопоточности

**Статус:** ⛔ ЗАПРЕЩЕНО.
`volatile` в C++ не гарантирует атомарность или порядок памяти. Используйте `std::atomic`.

> **Для понимания:** `volatile` в C++ говорит компилятору "не оптимизируй доступ к этой переменной". Это было нужно для
> memory-mapped I/O (регистры устройств), но НЕ для многопоточности. Для атомарного доступа нужны инструкции
> процессора (
`LOCK` префикс на x86), которые предоставляет `std::atomic`.
