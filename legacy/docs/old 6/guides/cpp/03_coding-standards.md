# Стандарты кодирования ProjectV

**🟢 Уровень 1: Начинающий** — Основные правила написания кода для ProjectV.

Стандарты кодирования в ProjectV обеспечивают предсказуемость, читаемость и высокую производительность кодовой базы.
Данные правила направлены на снижение когнитивной нагрузки при чтении кода и минимизацию ошибок.

---

## Обоснование стандартов

Применение единых стандартов необходимо для:

1. **Снижения когнитивной нагрузки:** единый стиль оформления позволяет сфокусироваться на логике, а не на форме.
2. **Единообразия кодовой базы:** код выглядит целостным независимо от автора.
3. **Эффективности ревью:** автоматизируемые правила оформления сокращают время на проверку кода.

> **Связь с философией:** Эти стандарты — прямое следствие принципов
> из [01_manifesto.md](../../philosophy/01_manifesto.md). Мы не вводим правила ради правил. Каждое ограничение имеет
> конкретную причину: производительность, предсказуемость или безопасность.

## Основные принципы

### 1. Использование современных стандартов C++

В проекте используется стандарт C++26. Новые возможности языка применяются только в случае их измеримой пользы:

- **Модули (Modules):** для изоляции интерфейсов и ускорения компиляции.
- **Концепты (Concepts):** для явного определения ограничений шаблонов.
- **Constexpr/Consteval:** для вычислений на этапе компиляции.
- **std::expected:** для типизированной обработки ошибок.

> **Для понимания:** C++26 — это не мода, а инструмент. Каждая новая фича должна отвечать на вопрос: "Как это делает код
> быстрее или безопаснее?" Например, `std::expected` заменяет исключения в hot paths, а `constexpr` переносит вычисления
> с
> runtime на compile-time.

### 2. Отказ от использования исключений (No Exceptions)

В ProjectV исключения отключены на уровне компилятора (`-fno-exceptions`) по следующим причинам:

- **Предсказуемость:** исключения затрудняют анализ путей выполнения кода.
- **Производительность:** обработка исключений вносит накладные расходы, недопустимые в критических участках.
- **Явность:** обработка ошибок через `std::expected` или коды возврата делает поток управления прозрачным.

> **Метафора:** Исключение — это `goto` с невидимой меткой. Когда вы бросаете `throw`, выполнение прыгает в неизвестное
> место (обработчик `catch`), минуя весь код между ними. В игровом движке это разрушает предсказуемость: деструкторы
> вызываются, стек разматывается, процессор теряет branch prediction. `std::expected<T, E>` из C++23 — это явный возврат
> либо значения, либо ошибки. Нет скрытых переходов.

```cpp
// ПЛОХО: Исключения (если бы они были включены)
VkBuffer createBuffer() {
    if (failed) throw VulkanError("Failed");  // Скрытый goto
    return buffer;
}

// ХОРОШО: std::expected
std::expected<VkBuffer, VulkanError> createBuffer() noexcept {
    if (failed) return std::unexpected(VulkanError("Failed"));
    return buffer;  // Явный путь выполнения
}
```

**Связь с философией:** См. [08_anti-patterns.md](../../philosophy/08_anti-patterns.md) — исключения в hot paths
считаются анти-паттерном.

### 3. Прагматичный подход к стандартной библиотеке (STL)

STL используется там, где это не вредит производительности. В критических участках (hot paths) STL-контейнеры могут быть
заменены на специализированные аллокаторы или структуры данных.

> **Для понимания:** STL спроектирована для общего случая. `std::vector` — отличный контейнер, но его `push_back` может
> вызвать реаллокацию в самый неподходящий момент. В hot path мы либо резервируем память заранее (`reserve()`), либо
> используем custom allocators с предвыделенной памятью.

**Связь с философией:** Подробнее о том, когда STL допустима, а когда нет —
в [05_optimization-philosophy.md](../../philosophy/05_optimization-philosophy.md).

### 4. Соглашения об именовании

- **snake_case:** для функций и переменных (`update_physics()`, `entity_count`).
- **PascalCase:** для типов и классов (`VoxelChunk`, `TransformComponent`).
- **Префиксы:** `m_` для членов классов, `s_` для статических переменных.

> **Для понимания:** Именование — это не эстетика, это коммуникация. Когда вы видите `m_transform`, вы сразу знаете: это
> член класса. Когда видите `update_physics()` — это функция. `VoxelChunk` — это тип. Единообразие снижает когнитивную
> нагрузку: мозг не тратит ресурсы на расшифровку, фокусируется на логике.

## Баланс между строгостью и гибкостью

Стандарты применяются максимально строго в:

- Публичных API.
- Основных системах движка (рендерер, физика, ECS).
- Коде, предназначенном для совместного использования.

В экспериментальном коде и прототипах допускаются временные отклонения от стандартов при условии последующего
рефакторинга.

> **Связь с философией:** Мы различаем "код-прототип" и "код-продакшн". Прототип пишется быстро, чтобы проверить идею.
> Продакшн пишется тщательно, чтобы работать годами.
> См. [10_decision-making.md](../../philosophy/10_decision-making.md).

## Управление стандартами

Стандарты кодирования являются живым документом. Они могут быть пересмотрены в следующих случаях:

1. Появление новых возможностей в следующих стандартах C++.
2. Обнаружение негативного влияния текущих правил на производительность.
3. Нахождение более эффективных паттернов проектирования.

---

## Критерии качества кода

1. **Читаемость:** код должен быть понятен специалисту без дополнительных пояснений.
2. **Поддерживаемость:** внесение изменений не должно приводить к неявным поломкам в других системах.
3. **Нулевые накладные расходы:** стандарты оформления не должны снижать производительность выполнения.

> **Связь с философией:** "Нулевые накладные расходы" — это принцип C++, описанный
> в [02_zero-cost-abstractions.md](../../philosophy/02_zero-cost-abstractions.md). Абстракция, которая стоит времени
> выполнения — это плохая абстракция для движка.

## Инструментарий

### clang-format

Файл `.clang-format` в корне проекта определяет автоматическое форматирование. Основные настройки:

- `BasedOnStyle: LLVM` — современная база
- `UseTab: Always` — табы для производительности
- `IndentWidth: 4` — баланс читаемости
- `PointerAlignment: Right` — стандарт C/C++ (`Type* ptr`)

### clang-tidy

Файл `.clang-tidy` настраивает статический анализ. Основные категории проверок:

- **Performance:** все проверки включены (критично для движка)
- **Bugprone:** большинство включено, исключения для графического кода
- **Modernize:** полезные (`use-emplace`, `use-nodiscard`, `use-noexcept`)

```bash
# Форматирование всех файлов
clang-format -i --style=file src/*.cpp

# Статический анализ
clang-tidy --config-file=.clang-tidy src/*.cpp --
```

## Запрещённые конструкции

Подробный список того, что нельзя использовать в ProjectV — в [11_banned-features.md](11_banned-features.md).

Краткий список:

| Конструкция                     | Причина запрета            | Альтернатива                 |
|---------------------------------|----------------------------|------------------------------|
| `new`/`delete`                  | Утечки, фрагментация       | Умные указатели, RAII        |
| Сырые указатели с владением     | Утечки                     | `unique_ptr`, `shared_ptr`   |
| Исключения                      | Непредсказуемость          | `std::expected`              |
| Глобальные переменные           | Состояние, race conditions | Dependency injection         |
| `using namespace std;`          | Конфликты имён             | Явные `std::`                |
| Виртуальные функции в hot paths | Cache miss, нет inline     | ECS, статический полиморфизм |
