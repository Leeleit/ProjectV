## 3. Структура репозитория

### 3.1 Основные ветви

| Ветвь       | Назначение                          | Защита                              |
|-------------|-------------------------------------|-------------------------------------|
| `main`      | Стабильная версия для продакшена    | Обязательный PR, требования к ревью |
| `develop`   | Интеграционная ветвь для разработки | Обязательный PR                     |
| `release/*` | Подготовка к релизу                 | Обязательный PR                     |
| `hotfix/*`  | Срочные исправления                 | Обязательный PR                     |

### 3.2 Именование ветвей

```
<тип>/<идентификатор>-<краткое-описание>

Примеры:
feature/PROJ-123-add-vulkan-renderer
bugfix/PROJ-456-fix-memory-leak
refactor/PROJ-789-optimize-ecs
docs/PROJ-012-update-cmake-guide
```

### 3.3 Типы ветвей

| Тип          | Префикс     | Назначение                               |
|--------------|-------------|------------------------------------------|
| Функционал   | `feature/`  | Новые возможности                        |
| Исправление  | `bugfix/`   | Исправление ошибок                       |
| Рефакторинг  | `refactor/` | Улучшение кода без изменения функционала |
| Документация | `docs/`     | Обновление документации                  |
| Релиз        | `release/`  | Подготовка к релизу                      |
| Хотфикс      | `hotfix/`   | Срочные исправления в продакшене         |

## 4. Стандарт коммитов

### 4.1 Формат сообщения коммита

```
<тип>(<область>): <описание>

[опциональное тело]

[опциональные метки]
```

### 4.2 Типы коммитов

| Тип        | Описание                              | Пример                                               |
|------------|---------------------------------------|------------------------------------------------------|
| `feat`     | Новая функциональность                | `feat(render): добавить поддержку mesh shaders`      |
| `fix`      | Исправление ошибки                    | `fix(memory): устранить утечку в PoolAllocator`      |
| `refactor` | Рефакторинг без изменения функционала | `refactor(ecs): оптимизировать итерацию компонентов` |
| `docs`     | Изменения документации                | `docs(cmake): обновить руководство по модулям`       |
| `style`    | Форматирование, стиль                 | `style: исправить отступы в core.cppm`               |
| `test`     | Добавление/изменение тестов           | `test(math): добавить тесты для Vec4`                |
| `chore`    | Обслуживание проекта                  | `chore: обновить .clang-format`                      |
| `perf`     | Оптимизация производительности        | `perf(voxel): оптимизировать генерацию чанков`       |

### 4.3 Области (scopes)

| Область   | Назначение                 |
|-----------|----------------------------|
| `core`    | Ядро движка                |
| `render`  | Подсистема рендеринга      |
| `vulkan`  | Vulkan-специфичное         |
| `physics` | Физика                     |
| `ecs`     | Entity Component System    |
| `ui`      | Пользовательский интерфейс |
| `voxel`   | Воксельная система         |
| `build`   | Система сборки (CMake)     |
| `docs`    | Документация               |

### 4.4 Примеры сообщений коммитов

```
feat(render): добавить поддержку VK_EXT_mesh_shader

Добавлена поддержка расширения VK_EXT_mesh_shader для
тесселяции на GPU. Включает:
- Создание pipeline для mesh shaders
- Интеграция с существующей системой рендеринга
- Unit-тесты для mesh shader pipeline

Closes: #123

BREAKING CHANGE: Требуется Vulkan 1.4+
```

```
fix(memory): устранить утечку в StackAllocator::rewind_to_marker

При откате к маркеру не освобождались временные буферы,
что приводило к утечке памяти в долгоживущих аллокаторах.

Fixes: PROJ-456
```

## 5. Процесс слияния (Merge Process)

### 5.1 Pull Request

Все изменения в защищённые ветви ДОЛЖНЫ проходить через Pull Request:

1. Создать ветвь от `develop` (или `main` для hotfix)
2. Внести изменения с соблюдением стандартов
3. Создать Pull Request в целевую ветвь
4. Пройти автоматические проверки (CI)
5. Получить одобрение минимум одного ревьюера
6. Выполнить squash-merge или rebase-merge

### 5.2 Требования к Pull Request

- [ ] Соответствие стандарту коммитов
- [ ] Прохождение CI-пайплайна (сборка, тесты, clang-format)
- [ ] Минимум одно одобрение от ревьюера
- [ ] Отсутствие конфликтов слияния
- [ ] Обновлённая документация (при необходимости)

### 5.3 Стратегия слияния

| Ветвь назначения | Стратегия      | Обоснование                       |
|------------------|----------------|-----------------------------------|
| `develop`        | Squash + Merge | Чистая история, один коммит на PR |
| `main`           | Merge commit   | Сохранение истории релизов        |
| `release/*`      | Rebase + Merge | Линейная история в релизной ветви |

## 6. Теги и релизы

### 6.1 Формат тегов

```
v<major>.<minor>.<patch>[-<prerelease>]

Примеры:
v0.1.0
v0.2.0-alpha.1
v1.0.0-rc.1
v1.0.0
```

### 6.2 Семантическое версионирование

| Изменение                                    | Увеличение версии |
|----------------------------------------------|-------------------|
| Обратно несовместимые изменения API          | Major             |
| Новая функциональность (обратно совместимая) | Minor             |
| Исправление ошибок                           | Patch             |

### 6.3 Аннотированные теги

```bash
# Создание аннотированного тега
git tag -a v1.0.0 -m "Релиз v1.0.0: Базовая функциональность"

# Отправка тега
git push origin v1.0.0
```

## 7. Git-хуки

### 7.1 Pre-commit

Автоматические проверки перед каждым коммитом:

```bash
#!/bin/bash
# .git/hooks/pre-commit

# Проверка форматирования
files=$(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(cpp|cppm|h)$')
if [ -n "$files" ]; then
    clang-format --dry-run --Werror $files
    if [ $? -ne 0 ]; then
        echo "ОШИБКА: Файлы не отформатированы. Выполните: clang-format -i $files"
        exit 1
    fi
fi

# Проверка clang-tidy
if [ -n "$files" ]; then
    clang-tidy -p build $files
    if [ $? -ne 0 ]; then
        echo "ОШИБКА: Обнаружены проблемы clang-tidy"
        exit 1
    fi
fi
```

### 7.2 Pre-push

Проверки перед отправкой в удалённый репозиторий:

```bash
#!/bin/bash
# .git/hooks/pre-push

# Запуск тестов
cmake --build build --target tests
./build/bin/tests

if [ $? -ne 0 ]; then
    echo "ОШИБКА: Тесты не прошли"
    exit 1
fi
```

## 9. Требования соответствия

### 9.1 Обязательные требования

1. Все коммиты ДОЛЖНЫ следовать формату Conventional Commits
2. Все изменения в защищённые ветви ДОЛЖНЫ проходить через Pull Request
3. Все Pull Request ДОЛЖНЫ иметь минимум одно одобрение
4. Все релизы ДОЛЖНЫ быть помечены аннотированными тегами

### 9.2 Запрещённые практики

1. Прямые коммиты в `main` и `develop`
2. Force-push в защищённые ветви
3. Удаление ветвей без слияния через PR
4. Коммиты с непроходящими тестами
