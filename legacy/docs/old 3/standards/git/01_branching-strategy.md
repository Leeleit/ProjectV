## 1. Область применения

Настоящий стандарт определяет стратегию ветвления Git для проекта ProjectV. Все операции создания и слияния ветвей
ДОЛЖНЫ соответствовать данной спецификации.

## 3. Модель ветвления

### 3.1 Диаграмма ветвления

```
main        ──●─────────────────────●────────────●──
             │                      ↑            ↑
release/*   │   ┌─────────●─────────┘            │
             │   │         ↑                     │
hotfix/*    │   │         │   ┌────────────────┘
             │   │         │   │
develop     ──●───●─●─●─●─●───●─●─●─●─●─●─●─●─●──●──
             │       ↑   ↑         ↑   ↑   ↑
feature/*   │   ┌───┘   └───┐ ┌───┘   └───┐
             │   │         │ │         │
            ──●─●─●─────●─●─●─●─────●─●─●──────────
```

### 3.2 Типы ветвей

| Тип         | Жизненный цикл | Источник  | Назначение             |
|-------------|----------------|-----------|------------------------|
| `main`      | Постоянная     | —         | Продакшн-код           |
| `develop`   | Постоянная     | `main`    | Интеграция разработки  |
| `feature/*` | Временная      | `develop` | Новая функциональность |
| `bugfix/*`  | Временная      | `develop` | Исправление ошибок     |
| `release/*` | Временная      | `develop` | Подготовка к релизу    |
| `hotfix/*`  | Временная      | `main`    | Срочные исправления    |

## 4. Постоянные ветви

### 4.1 Ветвь main

**Назначение:** Стабильная версия для продакшена.

**Правила:**

- Ветвь ДОЛЖНА всегда находиться в работоспособном состоянии
- Прямые коммиты ЗАПРЕЩЕНЫ
- Слияние только через Pull Request из `release/*` или `hotfix/*`
- Каждый коммит в `main` ДОЛЖЕН быть помечен тегом версии

**Защита:**

- Обязательный ревью: минимум 2 одобрения
- Обязательные проверки CI: сборка, тесты, статический анализ
- Запрет force-push
- Запрет удаления ветви

### 4.2 Ветвь develop

**Назначение:** Интеграционная ветвь для разработки.

**Правила:**

- Ветвь ДОЛЖНА быть стабильной
- Прямые коммиты ЗАПРЕЩЕНЫ
- Слияние из `feature/*`, `bugfix/*`, `refactor/*`

**Защита:**

- Обязательный ревью: минимум 1 одобрение
- Обязательные проверки CI: сборка, тесты
- Запрет force-push

## 5. Временные ветви

### 5.1 Feature-ветви

**Назначение:** Разработка новой функциональности.

**Жизненный цикл:**

1. Создать ветвь от `develop`
2. Разработать функциональность
3. Создать Pull Request в `develop`
4. После слияния — удалить ветвь

**Именование:**

```
feature/<идентификатор>-<краткое-описание>

Примеры:
feature/PROJ-123-vulkan-renderer
feature/PROJ-456-voxel-world-gen
feature/core-event-system
```

**Правила:**

- Одна функциональность — одна ветвь
- Ветвь ДОЛЖНА быть атомарной и слиться одним PR
- Ветвь ДОЛЖНА регулярно синхронизироваться с `develop`

### 5.2 Bugfix-ветви

**Назначение:** Исправление ошибок, обнаруженных в процессе разработки.

**Именование:**

```
bugfix/<идентификатор>-<краткое-описание>

Примеры:
bugfix/PROJ-789-memory-leak-pool
bugfix/vulkan-validation-errors
```

**Правила:**

- Создавать от `develop`
- Сливать обратно в `develop`

### 5.3 Refactor-ветви

**Назначение:** Рефакторинг без изменения функционала.

**Именование:**

```
refactor/<идентификатор>-<краткое-описание>

Примеры:
refactor/ecs-iteration-optimization
refactor/core-memory-layout
```

### 5.4 Release-ветви

**Назначение:** Подготовка к релизу.

**Жизненный цикл:**

1. Создать ветвь от `develop`
2. Заморозить функциональность
3. Исправить критические ошибки
4. Обновить версию и документацию
5. Слить в `main` и `develop`
6. Удалить ветвь

**Именование:**

```
release/<версия>

Примеры:
release/0.1.0
release/1.0.0
release/2.0.0
```

**Правила:**

- Новая функциональность ЗАПРЕЩЕНА
- Только исправления ошибок и обновление документации
- При слиянии в `main` создать тег версии

### 5.5 Hotfix-ветви

**Назначение:** Срочные исправления в продакшене.

**Жизненный цикл:**

1. Создать ветвь от `main` (соответствующего тега)
2. Исправить ошибку
3. Слить в `main` и `develop`
4. Создать патч-тег

**Именование:**

```
hotfix/<версия>-<краткое-описание>

Примеры:
hotfix/1.0.1-crash-fix
hotfix/1.0.2-vulkan-validation
```

## 6. Процесс работы с ветвями

### 6.1 Начало работы над функциональностью

```bash
# Переключиться на develop
git checkout develop

# Получить последние изменения
git pull origin develop

# Создать feature-ветвь
git checkout -b feature/PROJ-123-new-feature

# Выполнить работу...
git add .
git commit -m "feat(core): добавить новую функциональность"
```

### 6.2 Синхронизация с develop

```bash
# Получить последние изменения из develop
git fetch origin develop

# Rebase на develop
git rebase origin/develop

# Разрешить конфликты при наличии...
```

### 6.3 Создание Pull Request

```bash
# Отправить ветвь в remote
git push -u origin feature/PROJ-123-new-feature

# Создать Pull Request через веб-интерфейс или CLI
gh pr create --base develop --head feature/PROJ-123-new-feature \
    --title "feat(core): добавить новую функциональность" \
    --body "Описание изменений..."
```

### 6.4 После слияния

```bash
# Переключиться на develop
git checkout develop

# Получить изменения
git pull origin develop

# Удалить локальную ветвь
git branch -d feature/PROJ-123-new-feature

# Удалить remote-ветвь (если не удалена автоматически)
git push origin --delete feature/PROJ-123-new-feature
```

## 7. Разрешение конфликтов

### 7.1 Конфликты при rebase

```bash
# При возникновении конфликта
git status

# Разрешить конфликты вручную
# Открыть файлы с конфликтами в редакторе

# После разрешения
git add <разрешённые-файлы>
git rebase --continue

# Если нужно пропустить коммит
git rebase --skip

# Отмена rebase
git rebase --abort
```

### 7.2 Конфликты при слиянии PR

```bash
# Обновить ветвь относительно develop
git checkout feature/PROJ-123-new-feature
git fetch origin develop
git merge origin/develop

# Разрешить конфликты
# ...

# Завершить слияние
git add .
git commit -m "merge: разрешить конфликты с develop"

# Отправить изменения
git push origin feature/PROJ-123-new-feature
```

## 8. Требования соответствия

### 8.1 Обязательные требования

1. Все ветви ДОЛЖНЫ следовать соглашению об именовании
2. Все feature-ветви ДОЛЖНЫ создаваться от `develop`
3. Все hotfix-ветви ДОЛЖНЫ создаваться от `main`
4. Все ветви ДОЛЖНЫ быть удалены после слияния

### 8.2 Запрещённые практики

1. Создание ветвей от других feature-ветвей (без одобрения)
2. Долгоживущие feature-ветви (> 2 недель без синхронизации)
3. Прямые коммиты в `main` и `develop`
4. Force-push в любую ветвь без крайней необходимости
