# Стратегии работы с кэшем CPU (Caching Strategies)

В архитектуре ProjectV оптимизация использования кэша CPU является критическим фактором производительности. Основной
целью проектирования является обеспечение локальности данных (Cache Locality).

## 1. Пространственная локальность (Spatial Locality)

Обеспечение последовательного размещения данных в памяти минимизирует количество промахов кэша.

* **Эффективно:** Линейный обход `std::vector` или массивов.
* **Неэффективно:** Использование `std::list` или структур, где узлы распределены по куче.

```cpp
// Пример реализации (Spatial Locality)
void updatePhysics(Transform* t, Velocity* v, size_t count) {
    for (size_t i = 0; i < count; ++i) {
        // Данные t[i] и t[i+1] располагаются последовательно в кэш-линии
        t[i].pos += v[i].vel * dt;
    }
}
```

## 2. Предсказание ветвлений (Branch Prediction)

Оптимизация условий внутри циклов позволяет избежать сброса конвейера процессора при неверном предсказании ветвления.

```cpp
// Неэффективное ветвление в критическом цикле (Hot Loop)
for (auto& entity : entities) {
    if (entity.is_active) {
        doComplexMath(entity);
    }
}
```

**Решение:** Разделение активных и неактивных сущностей по разным массивам (или архетипам в Flecs) позволяет исключить
проверку условий внутри цикла.

### Атрибуты C++20

Использование атрибутов `[[likely]]` / `[[unlikely]]` для ветвлений с предсказуемым исходом:

```cpp
if (input == nullptr) [[unlikely]] {
    return; // Ошибка, маловероятный сценарий
}
```

## 3. Разделение горячих и холодных данных

Данные объектов разделяются на основе частоты их использования:

* **Hot (Горячие):** Позиция, скорость, состояние. Обрабатываются в каждом кадре.
* **Cold (Холодные):** Имена, описания, настройки ИИ. Используются эпизодически.

Рекомендуется разделение этих данных по разным структурам:

```cpp
// Горячий компонент (Hot Component)
struct Transform { glm::vec3 pos; };

// Холодный компонент (Cold Component)
struct EntityProfile { std::string name; std::string bio; };
```

Такой подход предотвращает загрузку неиспользуемых данных в кэш при выполнении критических вычислений.

## 4. Кэш инструкций (Instruction Cache / I-Cache)

Сложные функции с множеством ветвлений и виртуальными вызовами могут вызывать промахи кэша инструкций.

* Рекомендуется поддерживать компактность кода в критических циклах.
* Использование `inline` функций оправдано для небольших фрагментов кода, однако избыточный инлайнинг может привести к
  раздуванию бинарного кода и снижению эффективности I-Cache.

## 5. SoA vs AoS

Организация данных в памяти напрямую влияет на эффективность кэша. Детальный разбор с примерами для вокселей —
в [Memory Layout](02_memory-layout.md).

**Краткое правило:** `std::vector<Position>` вместо `std::vector<Entity>` — система работает только с нужными данными,
кэш не засоряется лишними полями.

## 6. Технические выводы

1. Использование массивов (`std::vector`, `std::array`) для обеспечения последовательного доступа.
2. Сортировка данных для минимизации прыжков по памяти.
3. Исключение сложных условий из циклов обработки сущностей.
4. Декомпозиция компонентов на независимые блоки данных (SoA) — подробнее в [Memory Layout](02_memory-layout.md).
5. Инструментальный контроль cache misses с использованием Tracy Profiler.
