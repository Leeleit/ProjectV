## 1. Правила выполнения плана

### 1.1. Базовые принципы

- Сначала рабочий вертикальный срез, потом расширение архитектуры.
- Не внедрять большие подсистемы "в стол" без визуального или функционального результата.
- Каждый этап должен заканчиваться runnable build.
- Каждый этап должен иметь чёткий критерий "готово".
- Любая новая подсистема сначала делается в минимальном варианте, затем расширяется.
- Оптимизацию выполнять только после появления корректного результата и базовых замеров.

### 1.2. Definition of Done для любой задачи

Задача считается завершённой только если:

- Код собирается в основном debug-профиле.
- Приложение запускается без немедленного падения.
- Нет новых ошибок validation layers или они объяснены и занесены в backlog.
- Есть логирование нештатных ситуаций.
- Обновлена документация, если затронут публичный модуль или workflow.
- Если задача влияет на производительность, сделан хотя бы базовый замер "до/после".

### 1.3. Приоритеты

- `P0` - критический путь проекта. Без этого движок не движется.
- `P1` - важно для следующего крупного этапа.
- `P2` - полезно, но можно отложить.
- `P3` - nice-to-have, пост-MVP или polish.

### 1.4. Технические принципы

- Основной язык: C++26.
- Основной API рендера: Vulkan 1.4.
- Для MVP на защиту приоритетен простой и предсказуемый путь: chunk rendering через graphics pipeline, камера,
  редактирование блоков, debug HUD.
- Compute shaders, ray marching, SVO, bindless и mesh shaders считаются направлением Vision/R&D и не должны блокировать
  MVP.
- Graphics pipeline использовать как основной путь MVP и как fallback даже после появления compute-экспериментов.
- DOD важнее "красивой ООП-иерархии".
- Flecs ECS входит в MVP, но внедряется минимально и по реальным сценариям, а не как пустой каркас "на будущее".
- Компоненты ECS хранят данные, системы содержат логику, глобальное состояние допускается через singleton-компоненты.
- Для высокочастотных систем предпочитать polling, для редких событий - observers/events.
- Для массовых горячих данных предпочитать SoA, для камеры, конфигов, debug UI и редких объектов допустим AoS.
- В hot path избегать исключений, RTTI, лишних аллокаций, pointer chasing и неявного shared ownership.
- Оптимизацию подтверждать замерами. Источник правды по производительности - Tracy и воспроизводимые benchmark-сцены.
- Любые тяжёлые зависимости подключать по фактической необходимости, а не "на будущее".

### 1.5. Архитектурные ограничения из `legacy/docs/latest`

- MVP проекта из legacy-документов - это фундамент движка для защиты, а не полная "игра мечты".
- Целевой MVP: `1-10` чанков, камера, редактирование/разрушение блоков, минимальный ECS, базовый physics
  raycast/collision, debug-only UI.
- Явно не включать в MVP: мультиплеер, AI/pathfinding, полноценное сохранение мира, аудио, сложный игровой UI,
  кроссплатформенность, SVO-оптимизацию как обязательный этап.
- Тестировать в первую очередь системы, инварианты и сложные алгоритмы; trivial code не раздувать тестами.
- GPU-код проверять validation layers, RenderDoc, smoke-сценами и ручной визуальной валидацией.
- Решения делить на обратимые и необратимые: если можно быстро поменять позже, не тратить на это неделю сейчас.

## 3. Критический путь проекта

### 3.1. Уточнение scope по `legacy/docs/latest`

Изначальный TODO слишком сильно тянул проект в сторону compute-first пути. Legacy-документы уточняют границы:

- Для защиты нужен не "идеальный будущий движок", а рабочий MVP.
- В MVP достаточно simple chunk rendering, камеры, Flecs, базового physics integration и разрушения блоков.
- Compute/ray marching/SVO допустимы как исследовательская ветка, но не как единственный путь к первой демонстрации.

### 3.2. Track A - MVP для защиты

Это главный путь, если цель - успеть сделать и защитить рабочий движок.

1. Стабилизировать текущую Vulkan-базу.
2. Добавить камеру и управление.
3. Описать voxel chunk на CPU.
4. Реализовать простой meshing path для чанка.
5. Загрузить chunk mesh на GPU через уже существующий buffer path.
6. Показать на экране первый видимый voxel chunk.
7. Добавить установку/удаление блока и перестроение чанка.
8. Интегрировать минимальный Flecs ECS вокруг реальных сущностей/систем.
9. Добавить базовый Jolt raycast/collision.
10. Добавить debug HUD, базовое профилирование и подготовить демо-сцену.

### 3.3. Track B - Vision / R&D

Этот путь делать только если Track A уже живой или если для него выделен отдельный человек и отдельная ветка.

1. Compute pipeline и storage image.
2. Compute-based voxel ray casting / ray marching.
3. GPU-driven culling.
4. LOD/SVO эксперименты.
5. Жидкости, CA, продвинутые эффекты.

### 3.4. Правило развилки

Если проект застрял, возвращаться к вопросу:
"Что минимально нужно, чтобы команда уже на этой неделе увидела и отредактировала первый voxel chunk?"

Если ответ "нужно ещё сначала написать SVO/mesh shaders/custom allocator", почти наверняка это scope creep и задачу
нужно перенести в Track B.

  же файлы.
- Один человек всегда назначается интегратором спринта: он не "главный начальник", а тот, кто следит, чтобы ветки
  сходились и проект запускался.
- Нельзя одновременно нескольким людям менять swapchain, frame loop и core renderer без явной договорённости.
- Любая задача должна заканчиваться runnable build или чётко оформленным draft PR с описанием, что ещё не готово.
- Каждая подсистема должна иметь владельца документации и smoke-сценария, а не только владельца кода.

### 26.2. Рекомендуемое разделение ролей

1. Интегратор / Build / CI / документация.
   Отвечает за `CMake`, профили сборки, общую сборку проекта, правила ветвления, smoke checklist, `README`, поддержку
   `TODO.md`, сведение веток в `develop`.
2. Vulkan foundation / frame loop.
   Отвечает за `swapchain`, `frame resources`, `pipeline`, устойчивость resize/shutdown, базовые рендер-пути и
   отсутствие validation errors.
3. Камера / input / debug UX.
   Отвечает за free-fly камеру, управление, захват мыши, debug hotkeys, crosshair, базовые overlay-элементы.
4. Воксельные данные / генерация мира.
   Отвечает за `VoxelChunk`, координаты, storage block ids/material ids, тестовую генерацию сцены и операции
   чтения/записи блока.
5. Meshing / GPU upload / chunk rendering.
   Отвечает за meshing чанка, конвертацию в `MeshCpu`, загрузку в GPU, обновление меша после изменения блока и рендер
   видимых чанков.
6. ECS / Physics / gameplay glue.
   Отвечает за минимальную интеграцию Flecs, сущности камеры/игрока/чанков, Jolt raycast/collision и связку "ввод ->
   действие -> изменение мира".

### 26.3. Как распределить работу по фазам

#### Фаза A. Первый видимый чанк

- Человек 1: доводит сборку, profiles, shader build, checklist запуска.
- Человек 2: стабилизирует renderer и готовит путь для chunk mesh draw.
- Человек 3: делает камеру и управление.
- Человек 4: делает chunk data model и генератор тестовой сцены.
- Человек 5: делает meshing и GPU upload.
- Человек 6: поднимает минимальный Flecs sandbox и исследует Jolt отдельно, не блокируя фазу A.

#### Фаза B. Редактирование мира и MVP demo

- Человек 1: собирает ветки, чинит интеграционные конфликты, готовит smoke tests.
- Человек 2: оптимизирует draw path и обработку обновления чанка.
- Человек 3: делает interaction layer: луч из камеры, выделение блока, UI-подсказки.
- Человек 4: реализует изменение данных чанка и dirty-region bookkeeping.
- Человек 5: перестраивает меш только у изменённых чанков.
- Человек 6: подключает Flecs к реальным сущностям и добавляет Jolt raycast/collision.

#### Фаза C. Полировка защиты

- Человек 1: документация, итоговый отчёт, сборка демо.
- Человек 2: validation, RenderDoc captures, стабильность.
- Человек 3: demo controls, HUD, запись видео/скриншотов.
- Человек 4: benchmark-сцены и world fixtures.
- Человек 5: performance pass по meshing/render path.
- Человек 6: physics demo, разрушение блоков, gameplay hooks.

### 26.4. Кого пускать в Track B

- Если у команды есть свободный ресурс, выделить одного человека на отдельную ветку `feature/compute-prototype`.
- Этот человек не должен ломать MVP-путь и не должен зависеть от незавершённых правок пяти остальных.
- Лучший кандидат для Track B - тот, кто уже стабильно понимает текущий Vulkan renderer и умеет сам диагностировать
  validation issues.

### 26.5. Как учиться по пути, а не тормозить проект

- Новичкам лучше владеть небольшим модулем целиком, чем "понемногу всем".
- Каждую сложную задачу сначала закрывать на маленьком примере: один чанк, один input action, один raycast.
- Раз в неделю делать внутреннее мини-демо друг другу, даже если готово очень мало.
- Раз в неделю менять reviewer-пары, чтобы знание проекта не запиралось в одном человеке.

## 27. Git и PR workflow для команды

### 27.1. Практический режим для MVP

Legacy-документы предлагают полноценный Git Flow. Для команды новичков на MVP использовать его упрощённый поднабор:

- `main` - только демонстрируемые и стабильные состояния.
- `develop` - основная интеграционная ветка.
- `feature/*` - новая функциональность.
- `bugfix/*` - исправления.
- `refactor/*` - безопасные структурные изменения.
- `release/*` и `hotfix/*` вводить только когда появится реальная релизная дисциплина.

### 27.2. Правила веток

- Не делать прямые коммиты в `main` и `develop`.
- Одна ветка = одна задача = один понятный PR.
- Желательно, чтобы ветка жила не дольше `2-4` дней.
- Перед открытием PR ветку синхронизировать с `develop`.
- Если задача разрослась, лучше разбить её на несколько PR, чем держать "вечную" ветку.

### 27.3. Правила PR

- Каждый PR должен содержать: что сделано, зачем, как проверено, что осталось рискованным.
- В `develop` мержить только после хотя бы одного ревью и локального smoke run.
- В `main` мержить только стабилизированное демо-состояние, по возможности после двух одобрений.
- Большие PR без запуска проекта не принимать, даже если "код вроде правильный".
- Если PR меняет архитектурный инвариант, автор обязан обновить документацию или `TODO.md`.

### 27.4. Размер и качество PR

- Целиться в PR до `400-600` строк осмысленных изменений, кроме сгенерированных файлов и шейдерных бинарников.
- Сначала findings и риски, потом красота кода.
- Если задача затрагивает hot path, автор обязан приложить хотя бы базовый замер или объяснить, почему он пока не нужен.
- Если задача затрагивает Vulkan, автор обязан перечислить manual checks: запуск, resize, shutdown, validation.

### 27.5. Ритм команды

- Ежедневно: короткий стендап `10-15` минут.
- Раз в `2-3` дня: интеграционное окно, когда ветки подтягиваются к `develop`.
- Раз в неделю: внутреннее демо и пересмотр приоритетов `P0/P1/P2`.
- На доске задач держать статусы: `Backlog`, `In Progress`, `Review`, `Blocked`, `Done`.

## 29. Контрольные вопросы перед началом любой новой крупной задачи

Перед тем как браться за новую подсистему, ответить письменно или устно на 5 вопросов:

1. Какой конкретный результат будет виден после завершения задачи?
2. Нужна ли эта задача сейчас для критического пути?
3. Что является минимальной версией реализации?
4. Как будет проверяться корректность?
5. Что эта задача может сломать в уже работающем проекте?

Если нет ответов на эти вопросы, задача либо преждевременна, либо слишком размыта.
