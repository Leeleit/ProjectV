# СТВ-GIT-003: Стандарт процесса Pull Request

---

## 1. Область применения

Настоящий стандарт определяет требования к процессу Pull Request в проекте ProjectV. Все запросы на слияние ДОЛЖНЫ
соответствовать данной спецификации.

---

## 3. Жизненный цикл Pull Request

### 3.1 Диаграмма состояний

```
┌─────────────┐
│   Черновик   │
│   (Draft)    │
└──────┬──────┘
       │ Готов к ревью
       ▼
┌─────────────┐     ┌─────────────┐
│   Открыт    │────▶│  На ревью   │
│   (Open)    │     │ (In Review) │
└──────┬──────┘     └──────┬──────┘
       │                   │
       │ Запрошены         │ Одобрено
       │ изменения         │
       ▼                   ▼
┌─────────────┐     ┌─────────────┐
│ Изменения   │     │  Одобрен    │
│ запрошены   │     │ (Approved)  │
└──────┬──────┘     └──────┬──────┘
       │                   │
       │ Внесены           │ Слияние
       │ изменения         │
       ▼                   ▼
┌─────────────┐     ┌─────────────┐
│   Открыт    │     │  Слит       │
│             │     │ (Merged)    │
└─────────────┘     └─────────────┘
```

### 3.2 Этапы процесса

1. **Создание:** Автор создаёт Pull Request
2. **Ревью:** Ревьюеры проверяют код
3. **Исправления:** Автор вносит изменения по замечаниям
4. **Одобрение:** Ревьюеры одобряют PR
5. **Слияние:** PR сливается в целевую ветвь

---

## 4. Требования к Pull Request

### 4.1 Шаблон описания

```markdown
## Описание

Краткое описание изменений и мотивация.

## Тип изменений

- [ ] Новая функциональность (feat)
- [ ] Исправление ошибки (fix)
- [ ] Рефакторинг (refactor)
- [ ] Документация (docs)
- [ ] Тесты (test)
- [ ] Оптимизация (perf)

## Связанные задачи

Closes #<номер-задачи>
Related to #<номер-задачи>

## Изменения

- Изменение 1
- Изменение 2
- ...

## Тестирование

Опишите, как тестировались изменения.

## Чек-лист автора

- [ ] Код соответствует стандартам проекта (clang-format, clang-tidy)
- [ ] Добавлены/обновлены тесты
- [ ] Обновлена документация
- [ ] Все тесты проходят
- [ ] Нет новых предупреждений компилятора

## Скриншоты (если применимо)

| До | После |
|----|-------|
|    |       |
```

### 4.2 Обязательные элементы

| Элемент          | Обязательность | Описание                       |
|------------------|----------------|--------------------------------|
| Заголовок        | Обязательно    | Соответствует формату коммитов |
| Описание         | Обязательно    | Краткое описание изменений     |
| Тип изменений    | Обязательно    | Выбран один из типов           |
| Связанные задачи | Рекомендуется  | Ссылка на issue/task           |
| Чек-лист         | Обязательно    | Все пункты отмечены            |
| Тестирование     | Рекомендуется  | Информация о тестировании      |

## 5. Процесс ревью

### 5.1 Требования к ревью

| Целевая ветвь | Минимум одобрений | Требования                 |
|---------------|-------------------|----------------------------|
| `main`        | 2                 | Полная проверка, все тесты |
| `develop`     | 1                 | Сборка и тесты             |
| `release/*`   | 2                 | Регрессионное тестирование |

### 5.2 Критерии ревью

**Код:**

- Соответствие стандартам C++26
- Правильное управление памятью (RAII)
- Обработка ошибок через `std::expected`
- Корректное выравнивание данных

**Архитектура:**

- Соответствие модульной структуре
- Изоляция внешних зависимостей (PIMPL)
- Отсутствие циклических зависимостей

**Производительность:**

- Отсутствие лишних аллокаций
- Эффективное использование SIMD
- Учет cache-line выравнивания

**Тестирование:**

- Unit-тесты для новой функциональности
- Пограничные случаи
- Обработка ошибок

### 5.3 Типы комментариев ревью

| Тип          | Обозначение | Действие              |
|--------------|-------------|-----------------------|
| ОБЯЗАТЕЛЬНО  | `MUST`      | Блокирует слияние     |
| РЕКОМЕНДАЦИЯ | `SHOULD`    | Желательно исправить  |
| ПРЕДЛОЖЕНИЕ  | `SUGGEST`   | Можно проигнорировать |
| ВОПРОС       | `QUESTION`  | Требуется разъяснение |
| ПОХВАЛА      | `PRAISE`    | Положительный отзыв   |

### 5.4 Примеры комментариев

```markdown
**MUST**: Нарушение стандарта RAII
> Здесь используется сырой new. Используйте std::make_unique.

**SHOULD**: Рекомендация по оптимизации
> Этот цикл можно заменить на std::ranges::transform.

**QUESTION**: Уточнение логики
> Почему выбран именно этот алгоритм хеширования?

**PRAISE**: Отличная реализация
> Хорошо структурированный код, легко читается.
```

## 6. CI/CD проверки

### 6.1 Автоматические проверки

| Проверка       | Условие       | Действие      |
|----------------|---------------|---------------|
| Сборка Debug   | Успех         | Обязательно   |
| Сборка Release | Успех         | Обязательно   |
| Unit-тесты     | 100% проходят | Обязательно   |
| clang-format   | Нет изменений | Обязательно   |
| clang-tidy     | Нет ошибок    | Обязательно   |
| Покрытие кода  | ≥ 80%         | Рекомендуется |

### 6.2 Конфигурация CI

```yaml
# .github/workflows/pr-check.yml
name: PR Check

on:
  pull_request:
    branches: [main, develop]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Установка Clang 18
        run: |
          sudo apt-get update
          sudo apt-get install -y clang-18 lld-18 libc++-18-dev libc++abi-18-dev

      - name: Конфигурация CMake
        run: |
          cmake -B build -GNinja \
            -DCMAKE_CXX_COMPILER=clang++-18 \
            -DCMAKE_BUILD_TYPE=Debug \
            -DENABLE_TESTS=ON

      - name: Сборка
        run: cmake --build build --parallel

      - name: Тесты
        run: ctest --test-dir build --output-on-failure

      - name: clang-format
        run: |
          clang-format-18 --dry-run --Werror \
            $(find src -name '*.cpp' -o -name '*.cppm' -o -name '*.h')

      - name: clang-tidy
        run: |
          clang-tidy-18 -p build \
            $(find src -name '*.cpp' -o -name '*.cppm')
```

## 7. Слияние Pull Request

### 7.1 Условия для слияния

- [ ] Все проверки CI пройдены
- [ ] Минимум требуемое количество одобрений
- [ ] Нет незавершённых обсуждений
- [ ] Нет конфликтов слияния
- [ ] Ветка актуальна относительно целевой

### 7.2 Стратегии слияния

| Стратегия      | Применение        | Результат          |
|----------------|-------------------|--------------------|
| Squash + Merge | Feature → develop | Один коммит        |
| Merge commit   | Release → main    | Сохранение истории |
| Rebase + Merge | Feature → release | Линейная история   |

### 7.3 Сообщение слияния

```
<тип>(<область>): <заголовок PR> (#<номер-PR>)

<тело PR>

Co-authored-by: Автор <author@example.com>
```

---

## 8. После слияния

### 8.1 Автоматические действия

1. Удаление ветви источника
2. Обновление связанных задач
3. Уведомление участников

### 8.2 Ручные действия

1. Локальное удаление ветви:
   ```bash
   git checkout develop
   git pull origin develop
   git branch -d feature/PROJ-123-new-feature
   ```

2. Обновление документации (при необходимости)

---

## 9. Требования соответствия

### 9.1 Обязательные требования

1. Все PR ДОЛЖНЫ использовать шаблон описания
2. Все PR ДОЛЖНЫ пройти проверки CI
3. Все PR ДОЛЖНЫ иметь минимум требуемое количество одобрений
4. Все обсуждения ДОЛЖНЫ быть разрешены до слияния

### 9.2 Запрещённые практики

1. Слияние без одобрения
2. Слияние с непройденными проверками CI
3. Игнорирование обязательных (MUST) замечаний ревью
4. Самостоятельное одобрение собственного PR
