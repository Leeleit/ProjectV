# Манифест: производительность — не опция, данные первее кода

Этот документ — философское ядро нашего движка. Он объясняет, почему мы принимаем те или иные технические решения. Если
ты знаешь C++, но хочешь понять, как выжать из железа максимум, — ты в нужном месте.

Мы не «оптимизируем потом». Мы с первого дня проектируем системы, которые уважают физику железа. Медленный код — это
архитектурная ошибка.

---

## Фундаментальные принципы

### 1. Производительность — это обязательство, а не опция

В мире, где у тебя **16 миллисекунд** на кадр, нет места «достаточно быстро». Каждая микросекунда на счету.

**Почему это важно для вокселей:**

- Один чанк = 16×16×16 = 4096 вокселей
- Видимая область = 10×10×10 чанков = 409 600 вокселей
- Каждый кадр = обработка миллионов элементов

Это невозможно без архитектуры, спроектированной под нагрузку с первого дня.

### 2. Данные прежде кода

Мы не начинаем с классов и методов. Мы начинаем с вопроса: **как эти данные лежат в памяти?**

> **OOP‑мышление:** «У меня есть объект, какие у него методы?»
> **Наше мышление:** «У меня есть массив данных, как мне его быстро обработать?»

Код — это инструмент трансформации данных, ничего больше. Если структура данных неоптимальна, самый гениальный алгоритм
будет тормозить.

### 3. Память — самое узкое место

Процессор может выполнить ~100 инструкций, пока данные идут из RAM. Память — это не быстрый жёсткий диск, это *
*библиотека в другом городе**.

```
L1 Cache  (32 KB)   — 4 цикла    — рабочий стол
L2 Cache  (256 KB)  — 12 циклов  — ящики стола
L3 Cache  (8 MB)    — 40 циклов  — шкаф в комнате
RAM       (16 GB+)  — 100+ циклов — библиотека в другом городе
```

> **Метафора:** Представь, что ты работаешь за столом. На столе (L1) лежат документы, которые ты используешь прямо
> сейчас. Ящики стола (L2) — документы, которые могут понадобиться. Шкаф в комнате (L3) — архив. А «библиотека в другом
> городе» (RAM) — это когда тебе нужно встать, сесть на автобус, ехать полчаса, найти нужную книгу, вернуться.
>
> Каждый `cache miss` — это поездка в библиотеку. Пока ты едешь — ты не работаешь. Процессор простаивает. Наша задача —
> сделать так, чтобы всё нужное лежало на столе.

Мы проектируем данные так, чтобы они помещались в L1/L2. Мы думаем о кэш‑линиях (64 байта) как об атомарных единицах.

### 4. Отказ от «оптимизируем потом»

Оптимизация — это не финальный этап, это **архитектурное решение**. Если код медленный, значит, мы не поняли задачу до
конца. Мы не маскируем плохую архитектуру микрооптимизациями.

## Принципы разработки

### Когда сомневаешься — используй brute force

Не преждевременно оптимизируй. Простой линейный поиск по массиву часто быстрее хитрой хеш‑таблицы на малых N.
Предсказатель ветвлений и кэш процессора любят простые циклы.

```cpp
// Плохо: преждевременная оптимизация
std::unordered_map<int, Voxel*> voxelLookup; // Hash overhead, cache misses

// Хорошо: brute force для малых наборов
std::vector<Voxel>& voxels; // Линейный проход, cache‑friendly
for (const auto& v : voxels) {
    if (v.id == targetId) return &v;
}
```

### Читаемость превыше умности

Код читается чаще, чем пишется. «Умный» код — это код, который будет непонятен через месяц.

```cpp
// Плохо: «умный» код
auto result = std::accumulate(v.begin(), v.end(), 0,
    [](int a, const auto& b) { return a + b.value * b.factor; });

// Хорошо: читаемый код
int result = 0;
for (const auto& item : v) {
    result += item.value * item.factor;
}
```

### Low Latency важнее Throughput

В играх важна не пропускная способность, а предсказуемость. Редкий кадр с задержкой 50 ms хуже, чем стабильные 16.6 ms.
Мы оптимизируем для **p99 latency**, не для среднего.

```cpp
// Плохо: высокий throughput, но непредсказуемые задержки
std::vector<Voxel> voxels; // reallocation = spike

// Хорошо: предсказуемая latency
std::vector<Voxel> voxels;
voxels.reserve(MAX_VOXELS); // Нет реаллокаций = нет спайков
```

### Cache Coherency — это религия

Процессор ждёт данные из RAM 100+ тактов. За это время можно выполнить 100+ инструкций. Память — это узкое место, а не
процессор.

Мы проектируем данные так, чтобы они помещались в L1/L2. Мы думаем о кэш‑линиях (64 байт) как об атомарных единицах.

### Знай своё железо до битов

Абстракции полезны, но ты должен понимать, что происходит «под капотом». Каждый `virtual` — это косвенный вызов. Каждый
`new` — это поиск в куче. Каждый cache miss — это 100+ потерянных тактов.

```cpp
// Ты должен знать, что это:
std::vector<Entity> entities;

// ...в памяти выглядит так:
// [Entity0][Entity1][Entity2]...
// И при итерации загрузится в кэш целиком

// А это:
std::vector<Entity*> entities;

// ...выглядит так:
// [ptr0][ptr1][ptr2]... -> [Entity0]  (где‑то в куче)
//                         -> [Entity1]  (в другом месте)
//                         -> [Entity2]  (ещё где‑то)
// И каждый разыменование = потенциальный cache miss
```

> **Метафора:** Разница между `vector<Entity>` и `vector<Entity*>` — это разница между «все книги на одной полке» и
> «каждая книга в разных домах». В первом случае ты идёшь вдоль полки и берёшь книгу за книгой. Во втором — бегаешь по
> городу. Указатель — это адрес. Адрес может указывать куда угодно. Случайный доступ по указателям = хаотичное блуждание
> по памяти = постоянные cache misses.

## Наш подход к разработке

### Код как гипотеза

Каждое архитектурное решение — это ставка: «если я расположу данные так, будет быстрее». Мы не гадаем — мы проверяем
профилировщиком (Tracy). Если гипотеза не подтвердилась — меняем код.

**Научный метод в программировании:**

1. **Наблюдение:** Tracy показывает hotspot
2. **Гипотеза:** «Если перейду на SoA, будет быстрее»
3. **Эксперимент:** Реализую, измеряю
4. **Вывод:** Принимаю или отвергаю изменение

### Техническая честность

C++ даёт нам доступ к железу, и мы должны понимать стоимость каждой абстракции:

| Конструкция              | Стоимость                              | Когда использовать               |
|--------------------------|----------------------------------------|----------------------------------|
| `virtual`                | Косвенный вызов + возможный cache miss | Интерфейсы с одним объектом      |
| `std::shared_ptr`        | Атомарные операции + control block     | Разделяемое владение (редко)     |
| `new`                    | Аллокация в куче                       | Только для долгоживущих объектов |
| `std::vector::push_back` | Возможная реаллокация                  | После `reserve()`                |
| `std::map`               | Дерево + косвенность                   | Когда нужна упорядоченность      |
| `std::unordered_map`     | Хеш + возможные коллизии               | Только для редких lookups        |

Не потому что мы их запрещаем — а потому что мы выбираем их осознанно.

> **Для понимания:** Это не значит, что `std::shared_ptr` — зло в принципе. Для UI‑кода, для менеджера ресурсов, для
> конфигурационных объектов — это отличный инструмент. Но если ты попытаешься обработать 400 000 вокселей в кадре, и
> каждый воксель будет держать `shared_ptr` на текстуру — ты убьёшь производительность атомарными операциями и cache
> misses. Контекст имеет значение. Hot path требует одного подхода, UI — другого.

### Учиться через архитектуру

Читая код нашего движка, разработчик должен понимать не только **что** он делает, но и **почему** он написан именно так.
Мы избегаем «магии» — всё должно быть объяснимо.

**Пример: Почему у нас нет базового класса Entity?**
Потому что `Entity` в ECS — это просто `uint64_t` (идентификатор). Нет данных — нет проблем с наследованием. Композиция
через компоненты — это не паттерн, это следствие понимания данных.
