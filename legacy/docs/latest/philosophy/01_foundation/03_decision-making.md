# Принятие технических решений: интуиция врёт в 70% случаев

В нашем движке технический выбор основывается на понимании последствий, а не на интуиции или модных тенденциях. Каждое
решение — это компромисс, и мы должны понимать, что именно мы меняем на что.

Интуиция программиста насчёт производительности ошибается в ~70% случаев. Tracy Profiler — наш главный аргумент.

---

## Критерии оценки

Когда перед нами несколько вариантов реализации, мы оцениваем их по пяти критериям:

**Производительность** — влияет ли это на frame time, cache misses, memory bandwidth? Это приоритет №1 для кода в
горячем пути.

**Простота** — можно ли реализовать проще без потери качества? Если «да» — более сложный вариант отклоняется. Сложность
должна быть оправдана.

**Сопровождаемость** — поймёт ли другой разработчик этот код без дополнительных пояснений через полгода?

**Гибкость** — не закрывает ли это решение дверь для будущих улучшений? Публичные API и форматы данных требуют особой
осторожности — их менять дорого.

**Соответствие архитектуре** — работает ли это в парадигме DOD+ECS, или это исключение, которое тянет нас к
ООП‑мышлению?

## Как мы принимаем решения

### Обратимые vs необратимые

Прежде чем погружаться в анализ — задай вопрос: насколько легко это решение отменить?

- **Необратимые** (публичные API, форматы хранения данных, выбор библиотек‑зависимостей): требуют прототипа и замеров.
- **Обратимые** (внутренняя реализация, алгоритм в одной системе): делай, смотри на результат, рефакторь при
  необходимости.

Не трать два дня на анализ обратимого решения.

> **Метафора:** Представь, что ты выбираешь мебель. Диван — обратимое решение. Купил, не понравился — продал, купил
> другой. Хлопотно, но возможно. Фундамент дома — необратимое решение. Залил неправильно — дом развалится, переделывать
> дорого. На диван можно потратить 10 минут размышлений. На фундамент — дни расчётов и консультаций.

### Измеряй, а не догадывайся

Если между двумя вариантами стоит вопрос производительности — прототипируй и мерь. «Кажется, что быстрее» — не аргумент.
Tracy показывает цифры, цифры не врут.

Порог: если прирост производительности меньше 5–10% при значительном усложнении кода — выбираем простой вариант.

> **Для понимания:** «Очевидно, что хеш‑таблица быстрее линейного поиска» — но на малых N линейный поиск быстрее из‑за
> кэша. «Очевидно, что SIMD даёт 4x ускорение» — но только если данные выровнены и доступны последовательно.
> Единственный
> способ узнать правду — измерить.

### Врач не режет без УЗИ

Мы не оптимизируем без Tracy. Профайлер — это наш «УЗИ». Он показывает, где болит, а где здорово. Не лечи здорового и не
калечь больного.

> **Метафора:** Пациент говорит: «У меня болит живот, наверное аппендицит». Врач не режет сразу. Он делает анализы, УЗИ,
> собирает анамнез. Может, это просто несварение. Точно так же с оптимизацией: «Мне кажется, этот код медленный» — не
> повод переписывать. Сначала Tracy, потом решение.

### YAGNI и поэтапная абстракция

Не разрабатывай функциональность «на будущее». Абстрагируй код только тогда, когда у тебя есть минимум две разные
конкретные реализации. До этого — прямое решение без обёрток.

> **Для понимания:** YAGNI = «You Aren't Gonna Need It». Программисты любят строить «гибкие» системы «на будущее». Но
> будущее непредсказуемо. 90% заготовок никогда не используются, а поддерживать их приходится. Лучшее решение — писать
> код
> под конкретную задачу. Когда появится вторая задача — обобщить. Не раньше.

### Локальное vs глобальное

Решение может быть локально эффективным, но глобально вредным. Пример: кэш в одной системе ускоряет её, но усложняет
синхронизацию во всём движке. Всегда смотри на систему в целом.

## Документирование выбора

Нетривиальные архитектурные решения фиксируются — либо в документации, либо в комментарии в коде:

```cpp
// Используем flat array вместо std::unordered_map: бенчмарк показал
// 3x ускорение на 1000 чанков из‑за cache locality.
// Tracy: map → 2.1 ms, array → 0.7 ms (тест: optimization/chunk_lookup_bench.cpp)
```

Это экономит время при следующем ревью и предотвращает «почему так написано — давайте перепишем».

> **Для понимания:** Комментарий «почему» важнее комментария «что». Код показывает *что* он делает. Но *почему* он
> делает именно так — это контекст, который теряется со временем. Через полгода ты сам не вспомнишь, почему выбрал array
> вместо map. Документируй решение — сэкономишь время себе и другим.

## Пример: Выбор контейнера для чанков

**Задача:** Хранить чанки мира для быстрого доступа по координатам.

**Варианты:**

1. `std::unordered_map<Coord, Chunk*>` — O(1) lookup
2. `std::map<Coord, Chunk*>` — O(log N) lookup, упорядочен
3. `flat_map<Coord, Chunk*>` (сортированный вектор) — O(log N) lookup, cache‑friendly
4. Фиксированный 3D массив — O(1) lookup, предсказуемая память

**Анализ:**

| Вариант       | Lookup   | Cache   | Память | Сложность |
|---------------|----------|---------|--------|-----------|
| unordered_map | O(1)     | Плохо   | Выше   | Низкая    |
| map           | O(log N) | Плохо   | Выше   | Низкая    |
| flat_map      | O(log N) | Хорошо  | Низкая | Средняя   |
| 3D массив     | O(1)     | Отлично | Фикс.  | Низкая    |

**Решение:** Для активной области вокруг игрока — 3D массив (предсказуемый размер, идеальный cache locality). Для всего
мира — flat_map (компактность, cache‑friendly итерация).

**Почему не unordered_map:** Node‑based контейнер разбрасывает чанки по памяти. Каждый lookup — потенциальный cache
miss. Для 1000 чанков разница между O(1) и O(log N) незаметна (~10 vs ~20 сравнений), а cache locality даёт 3x
ускорение.

## Правило 80/20 для оптимизации

80% времени программа проводит в 20% кода. Не оптимизируй всё подряд. Сначала найди эти 20% (через Tracy), потом бей
точно в цель.

> **Метафора:** Представь, что ты стрелок. У тебя 100 патронов. Можно стрелять наугад по всей мишени, надеясь попасть в
> центр. А можно прицелиться в яблочко и сделать один точный выстрел. Tracy — это прицел. Он показывает, куда целиться.
