# Философия «Злых Хаков»: легальное зло в C++26

Иногда «правильный» код (Clean Code) работает медленно. Иногда **грязный** код (Hack) работает быстро. В нашем движке мы
прагматики. Если хак даёт +20% FPS и изолирован — мы его берём.

Это не оправдание плохого кода. Это осознанный выбор в пользу производительности, когда альтернативы нет.

---

## Легальное Зло (Legal Evil)

### 1. `std::bit_cast` (C++20)

Раньше мы использовали `union` или `reinterpret_cast` для type punning (превратить `float` в `int` для побитовых
операций). Это было UB (Undefined Behavior). Теперь у нас есть `std::bit_cast`. Это **легальный** способ смотреть на
биты.

```cpp
float f = 123.456f;
uint32_t bits = std::bit_cast<uint32_t>(f);
// Теперь можно хакать экспоненту
```

> **Для понимания:** Type punning — это когда ты смотришь на одни и те же биты как на разные типы. `float` и `int` — оба
> 4 байта. Но если ты попытаешься прочитать `float` как `int` через `reinterpret_cast`, это UB — компилятор имеет право
> сделать что угодно. `std::bit_cast` — это легальный способ сказать компилятору: «я знаю, что делаю, просто скопируй
> биты».

### 2. Manual SIMD (Intrinsics)

Компилятор (Auto‑vectorization) глупый. Он не знает контекста. В критических местах (математика вокселей, физика) мы
пишем SIMD вручную или через обёртки (`xsimd`, `highway`). Использование AVX2 / AVX‑512 напрямую — это норма для ядра
движка.

> **Для понимания:** SIMD (Single Instruction, Multiple Data) — это когда одна инструкция обрабатывает 4, 8 или 16 чисел
> одновременно. Вместо 4 умножений — одно. Авто‑векторизация компилятора работает в простых случаях, но ломается на
> сложных циклах. Ручной SIMD даёт гарантию: ты контролируешь, что процессор делает. Это сложнее, но предсказуемее.

### 3. Custom Allocators & No Bounds Checks

В Release‑сборке `std::vector::operator[]` не проверяет границы. Мы идём дальше. Мы пишем свои контейнеры, которые *
*никогда** не проверяют границы и не инициализируют память нулями при создании.

* `PODVector`: Массив `T`, который не вызывает конструкторы/деструкторы, а просто делает `malloc`/`free`. (Для
  тривиальных типов).

> **Для понимания:** `std::vector` инициализирует элементы при `resize()`. Это защита от неинициализированной памяти, но
> это время. Если ты всё равно перезапишешь все элементы — инициализация была впустую. `PODVector` для POD‑типов (int,
> float, простые struct) пропускает инициализацию. Опасно? Да, если забудешь записать. Быстро? Да, на миллионах
> элементов
> разница заметна.

### 4. Assumption Hints (`[[assume]]`)

Мы подсказываем оптимизатору «факты», которые он не может доказать сам.

```cpp
void process(int* data, int count) {
#ifdef __clang__
    __builtin_assume(count > 0 && count % 16 == 0); // Разверни цикл без проверок!
#endif
    // ...
}
```

В C++23 используем `[[assume(...)]]`.

> **Для понимания:** Компилятор не может знать то, что знаешь ты. Если `count` всегда кратен 16, компилятор не будет это
> предполагать — вдруг кто‑то передаст 17? `__builtin_assume` говорит: «поверь мне, это всегда правда». Компилятор
> использует это для оптимизации: разворачивает циклы, убирает проверки. Но если ты соврал — UB.

## Когда нельзя?

1. **В публичном API.** Пользователь не должен страдать от нашего безумия.
2. **В коде геймплея.** Скрипты должны быть безопасными.
3. **Без комментария `// EVIL:`.** Каждый хак должен быть помечен.

> **Для понимания:** Хак — это технический долг. Он работает сейчас, но может сломаться при смене компилятора, платформы
> или стандарта. Комментарий `// EVIL:` — это предупреждение: «здесь живёт опасный код, не трогай без понимания».
> Документируй хак, объясни, почему он нужен, что он делает, и какие есть альтернативы.

## Пример: Evil Hack в нашем движке

```cpp
// EVIL: SIMD intrinsics для быстрого вычисления обратного квадратного корня
// Используется в: генерация ландшафта (миллионы вызовов)
// Альтернатива: std::sqrt (медленнее в 8x на AVX2)
// Риск: Код привязан к x86 SSE/AVX, требует #ifdef для других платформ
// Тест: benchmarks/fast_sqrt_bench.cpp
#include <immintrin.h>

// Обрабатывает 8 float одновременно на AVX2
inline void fast_inv_sqrt_batch_avx2(const float* input, float* output, size_t count) {
    for (size_t i = 0; i < count; i += 8) {
        __m256 x = _mm256_loadu_ps(input + i);
        __m256 rsqrt = _mm256_rsqrt_ps(x);  // Аппаратная инструкция rsqrtps
        _mm256_storeu_ps(output + i, rsqrt);
    }
}

// Fallback для скалярного кода (ARM, старые CPU)
inline float fast_inv_sqrt_scalar(float x) {
    return 1.0f / std::sqrt(x);  // Компилятор оптимизирует в rsqrtss на x86
}
```

> **Почему не Quake III hack:** Знаменитый `0x5F3759DF` был актуален в 1999 году, когда CPU не имели аппаратных
> инструкций для rsqrt. На современных x86/ARM инструкции `rsqrtss`/`rsqrtps`/`vrsqrtps` работают быстрее и дают лучшую
> точность. Битовые хаки с float — это история, не современная практика.

## Правила использования хаков

### 1. Изолируй хак

Хак должен жить в отдельном файле (`fast_math.hpp`), а не размазан по коду. Легче заменить один файл, чем искать все
вхождения.

### 2. Документируй риски

Что сломается, если хак перестанет работать? На каких платформах? Какие есть fallback?

### 3. Имей fallback

Всегда имей безопасную, медленную реализацию на случай, если хак не поддерживается (ARM, старые CPU).

### 4. Тестируй на всех платформах

Хак, работающий на x86, может сломать сборку под ARM. CI должен прогонять тесты на всех целевых платформах.

---

## Почему мы это делаем?

Потому что в high‑performance движке **каждая микросекунда на счету**. Если хак даёт +20% FPS в критическом пути — это
не «грязный код», это **инженерное решение**.

Но помни: хак — это последнее средство, а не первое. Сначала оптимизируй данные, алгоритмы, архитектуру. Только потом —
микрооптимизации и хаки.

> **Метафора:** Представь гонщика. Он может ехать на серийном автомобиле — безопасно, предсказуемо. Но чтобы выиграть
> гонку, он модифицирует двигатель, снимает лишний вес, жертвует комфортом. Это риск: машина может сломаться. Но если он
> знает, что делает, и имеет запасной двигатель — это оправданный риск. Хаки — это модификации для гонки. Не для
> повседневной езды.
