# Анти‑паттерны: что нельзя использовать из стандартного C++

В этом документе перечислены технические решения, которые запрещены или не рекомендуются в нашем движке. Мы пишем
High‑Performance Real‑Time Engine, а не Enterprise Backend. Здесь железо диктует правила, а не удобство программиста.

---

## Анти‑паттерны C++

### 1. STL в Hot Path

Стандартная библиотека (STL) универсальна, но не оптимальна для критических путей.

* **`std::vector`:** В Debug ужасно медленный (проверки итераторов). В Release аллокатор общего назначения может быть
  медленным.
* **`std::shared_ptr`:** Атомарный счетчик ссылок убивает масштабируемость на многопоточке. Используйте `unique_ptr` или
  сырые указатели (если владение очевидно).
* **`std::map` / `std::unordered_map`:** Node‑based контейнеры — убийцы кэша. Используйте `flat_map` (сортированный
  вектор) или Open Addressing Hash Maps (например, `robin_hood` hashing).

> **Для понимания:** STL спроектирована для общего случая. `std::unordered_map` работает хорошо для 100–1000 элементов в
> enterprise‑приложении. Но в GameDev у нас другие требования: предсказуемая latency, отсутствие аллокаций в hot path,
> cache locality. Node‑based контейнеры (каждый элемент — отдельная аллокация) разбрасывают данные по памяти. Для UI или
> конфигурации — нормально. Для обработки 100 000 сущностей в кадре — убийство производительности.

### 2. Exceptions (Исключения)

Мы **не используем** исключения (`try / throw / catch`) в Runtime.

* Оверхед на размер бинарника (RTTI, Unwind Tables).
* Непредсказуемое время исполнения (нарушение Real‑Time).
* **Альтернатива:** Коды ошибок, `std::expected` (C++23), Assertions для фатальных ошибок.

> **Для понимания:** Исключения в C++ — это «zero‑cost» только если они не бросаются. Но таблицы unwind всё равно
> занимают место в бинарнике, а сама механика throw/catch тяжёлая. В игре, где 60 FPS = 16.6 ms на кадр, непредсказуемый
> throw может вызвать лаг. Для серверных приложений это нормально — там задержка в 10 ms незаметна. Для игр — это
> stutter.

### 3. RTTI (`dynamic_cast`, `typeid`)

Если вам нужно узнать тип объекта в рантайме — ваша архитектура сломана.

* `dynamic_cast` медленный (проход по дереву наследования).
* **Альтернатива:** ECS (компоненты определяют поведение), Tag Dispatching, `enum TypeID`.

## Анти‑паттерны Производительности

### 4. Pointer Chasing (Погоня за указателями)

`Object → Component → Data → Value`

Каждая стрелочка — это потенциальный Cache Miss (100–300 тактов CPU).

**Решение:** Data‑Oriented Design. Данные лежат в плоских массивах.

> **Метафора:** Представь, что ты работаешь в огромном архиве. Тебе нужно найти документ. В подходе с указателями ты
> смотришь в каталог: «Документ А находится в комнате 42». Идёшь в комнату 42. Там записка: «Документ А перенесён в
> комнату 17». Идёшь в комнату 17. Там ещё одна записка… Каждая записка — это указатель. Каждый переход — это время.
>
> **Pointer Chasing** — это когда код следует по цепочке указателей. `entity→component→data→field`. Четыре прыжка =
> четыре потенциальных cache miss. Если данные разбросаны по памяти — процессор spends 90% времени в ожидании.
>
> **Решение DOD:** Все данные лежат в одном массиве. Ты берёшь документ, и сразу получаешь все нужные поля. Никаких
> записок, никаких переходов.

### 5. False Sharing

Размещение атомиков или часто изменяемых данных разных потоков в одной кэш‑линии (64 байта).

**Решение:** `alignas(64)` для структур, используемых в разных потоках.

> **Метафора:** Представь два писателя за одним маленьким столом. Оба хотят писать. Но стол один. Первый взял ручку —
> второй ждёт. Второй взял блокнот — первый ждёт. Это false sharing: данные разные, но они делят одну кэш‑линию (стол).
> Когда один поток пишет в своё поле, вся кэш‑линия инвалидируется у других ядер. Координация через кэш‑линию — это
> неявный mutex.
>
> **Решение:** Разнести данные разных потоков на разные «столы» (кэш‑линии). `alignas(64)` гарантирует, что каждая
> структура займёт свою кэш‑линию целиком.

### 6. «Zero‑Copy» без замеров

Иногда скопировать 64 байта быстрее, чем передать их по ссылке (разыменование указателя). Не бойтесь копировать
маленькие структуры (`vec3`, `Matrix4x4`).

> **Для понимания:** Передача по ссылке — это передача указателя. Чтобы прочитать данные, нужно разыменовать указатель.
> Это потенциальный cache miss. Копирование 16–64 байт — это просто запись в регистры или стек, без обращения к памяти.
> Для маленьких структур копирование часто быстрее.

### 7. Mutex как туалет с замком

`std::mutex` — это «туалет с замком». Один зашёл, закрыл дверь, остальные ждут. Пока ждут — они ничего не делают. В Job
System мы не ждём. Если туалет занят — идём работать в другой комнате.

**Решение:** Lock‑free структуры данных, Work Stealing, Double Buffering.

## Анти‑паттерны Архитектуры

### 8. Singleton Abuse

Синглтоны скрывают зависимости и порядок инициализации.

**Решение:** Explicit Dependency Injection. Передавайте `World&`, `Renderer&` в функции обновления.

> **Для понимания:** Синглтон кажется удобным: «Я могу обратиться к `Game::instance()` из любой точки кода». Но это
> создаёт неявные зависимости. Функция `foo()` вызывает `Game::instance()` — но из её сигнатуры этого не видно.
> Тестировать такую функцию невозможно без инициализации всего движка. Dependency Injection делает зависимости явными:
`foo(Game& game)` показывает, что функции нужно.

### 9. God Object (Всемогущий объект)

Класс `Game` или `Player`, в котором 5000 строк кода.

**Решение:** ECS. Разбивайте состояние на компоненты, поведение на системы.

### 10. Premature Abstraction

«А вдруг нам понадобится сменить рендер с Vulkan на ASCII в консоли?»

Не понадобится. Пишите код для конкретной задачи. Абстракции добавляйте только тогда, когда появляется **вторая**
реализация.

> **Для понимания:** YAGNI (You Aren't Gonna Need It). Абстракция не бесплатна: она добавляет уровень косвенности,
> усложняет код, затрудняет оптимизацию. Не создавай абстракцию «на будущее». Создавай её когда у тебя уже есть две
> конкретные реализации, и ты видишь общий паттерн. До этого — прямой код без обёрток.

## Контекст имеет значение

> **Важно:** Все эти анти‑паттерны — это проблемы только в **hot path**. Для UI‑кода, для конфигурации, для инструментов
> редактора — используй `std::shared_ptr`, `std::map`, exceptions, синглтоны. Это нормально. Не преждевременно
> оптимизируй
> код, который не влияет на FPS.
>
> Проблема начинается, когда привычки из «обычного» кода переносятся в hot path. `std::shared_ptr` в UI — это хорошо.
`std::shared_ptr` на каждый из 400 000 вокселей — это смерть. Контекст имеет значение.
