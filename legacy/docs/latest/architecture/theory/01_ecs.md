# ECS: Основные концепции

Entity Component System (ECS) — это архитектура, разделяющая данные, поведение и идентичность объектов. В ProjectV мы
используем библиотеку **Flecs**.

## 1. Сущности (Entities)

Сущность — это просто **ID** (`uint64_t`). Это не объект в памяти, не класс, не структура.
Это просто число, которое говорит "Я существую".

* **Пример:** `Entity 42` (Игрок). `Entity 105` (Меч).
* **Свойство:** К сущности можно прикрепить (или открепить) компоненты в любой момент.

## 2. Компоненты (Components)

Компоненты — это **Данные** (`struct`).
Они не имеют методов (кроме конструкторов и простых геттеров).
Они не знают о других компонентах.

### Правила ProjectV:

* **POD-типы (Plain Old Data):** `float`, `int`, `vec3`.
* **Минимум `std::string`:** Строки — это аллокации. Используйте `const char*` или строковые ID, если возможно.
* **Нет логики:** В компоненте `Position` не должно быть метода `move()`.

```cpp
struct Position { float x, y, z; };
struct Velocity { float x, y, z; };
struct Name { const char* value; };
```

## 3. Системы (Systems)

Системы — это **Логика** (Функции).
Они обрабатывают сущности, у которых есть *определенный набор* компонентов.

### Query-Based Logic

Система говорит миру: *"Дай мне все сущности, у которых есть `Position` И `Velocity`"*.
Мир (World) дает ей итератор. Система бежит по итератору и обновляет данные.

```cpp
// Пример (псевдокод)
void MoveSystem(iter& it) {
    auto p = it.term<Position>(1);
    auto v = it.term<Velocity>(2);

    for (auto i : it) {
        p[i].x += v[i].x * dt;
        p[i].y += v[i].y * dt;
        p[i].z += v[i].z * dt;
    }
}
```

## 4. Мир (World)

Мир — это **База Данных**.
Он хранит все сущности, все компоненты и все системы.
Он управляет памятью (выделяет, освобождает).
Он запускает системы (Pipeline).

### Порядок выполнения

В ECS порядок систем важен.

1. **InputSystem**: Читает ввод.
2. **PhysicsSystem**: Обновляет позиции.
3. **CameraSystem**: Обновляет матрицу вида.
4. **RenderSystem**: Рисует мир.

## Преимущества для Вокселей

1. **Масштабируемость:** Можно добавить 100 000 частиц. Система `ParticleSystem` просто обработает их в цикле. Никаких
   `virtual update()` для каждой частицы.
2. **Кэш-локальность:** Данные частиц лежат массивом. Процессор счастлив.
3. **Гибкость:** Хотите, чтобы воксель падал? Добавьте ему компонент `Gravity`. Не надо наследовать `FallingVoxel` от
   `Voxel`.

# Взаимодействие систем в ECS

В ProjectV системы проектируются как независимые модули обработки данных. Это упрощает расширение функциональности и
обеспечивает возможность параллельного выполнения систем на многоядерных процессорах.

## 1. Синглтоны (Singleton Components)

Глобальные данные, доступные для всех систем (например, `Time`, `Input`, `GameConfig`), хранятся в ECS как
компоненты-синглтоны (компоненты, существующие в единственном экземпляре в рамках мира).

```cpp
// Определение глобальных данных
struct Time {
    float dt;
    float elapsed;
};

// Доступ из любой системы:
const Time* t = world.get<Time>();
float current_dt = t->dt;
```

Преимущество использования синглтонов заключается в отсутствии явных зависимостей между системами, так как они
взаимодействуют через общее состояние мира.

## 2. События и Обсерверы (Events & Observers)

Для выполнения разовых действий при изменении состояния сущностей используются обсерверы (Observers). Это позволяет
избежать проверки условий в каждом кадре для систем с редкими событиями.

### Типы событий:

- `OnAdd`: Срабатывает при добавлении компонента к сущности.
- `OnSet`: Срабатывает при изменении данных существующего компонента.
- `OnRemove`: Срабатывает непосредственно перед удалением компонента.

```cpp
// Реактивная логика через обсервер
world.observer<Health>()
    .event(flecs::OnSet) // Срабатывает при изменении HP
    .each( {
        if (h.current <= 0.0f) {
            it.entity(i).add<DeadTag>(); // Добавление тега смерти
            playDeathSound(it.entity(i)); // Проигрывание звука
        }
    });
```

Использование обсерверов снижает нагрузку на CPU, так как код выполняется только при фактическом изменении данных.

## 3. Выбор метода взаимодействия: Опрос (Polling) против Событий

Для систем с высокой частотой обновления и большим объемом данных предпочтительным является метод опроса (Polling), в то
время как для редких изменений используются события.

### Опрос (Polling):

- Применяется в системах физики, рендеринга и воксельного мешинга.
- Обеспечивает предсказуемую и стабильную нагрузку на кадр.
- Реализуется через стандартный механизм `world.progress()`.

### События (Events):

- Применяются для пользовательского интерфейса (UI), триггеров и редких игровых событий.
- Позволяют снизить среднюю нагрузку, но могут приводить к пиковым всплескам при одновременном срабатывании множества
  событий.

## 4. Управление межсистемными зависимостями

Для передачи результатов вычислений между системами в рамках одного кадра используются промежуточные компоненты (
Intermediate Components).

### Механизм взаимодействия:

1. Система А добавляет промежуточный компонент (например, `CollisionResult`) к сущности.
2. Система Б в этом же кадре считывает данные из `CollisionResult`, выполняет необходимые действия и удаляет компонент.

Данный подход обеспечивает слабую связанность (loose coupling) систем и упрощает их изолированное тестирование. Принципы
организации данных описаны в [Memory Layout](02_memory-layout.md).

# Композиция сущностей в ECS

В ProjectV проектирование объектов базируется на принципе композиции компонентов вместо иерархического наследования.
Свойства и поведение сущности определяются набором ассоциированных с ней данных (компонентов).

## 1. Теги (Tags)

Теги представляют собой компоненты без данных (пустые структуры), используемые для маркировки сущностей и фильтрации в
системах.

```cpp
// Определения тегов
struct Enemy {};
struct Player {};
struct Visible {};

// Использование в Flecs:
world.entity("Orc")
    .add<Enemy>()
    .add<Visible>();
```

Использование тегов позволяет динамически изменять поведение сущности во время выполнения. Например, удаление тега
`Visible` исключает сущность из обработки системой рендеринга без влияния на другие системы (физика, ИИ).

## 2. Префабы (Prefabs)

Префабы являются шаблонами сущностей, которые хранят предопределенный набор компонентов и их значений. Они используются
для эффективного создания множества однотипных объектов.

```cpp
// Создание префаба
auto GoblinPrefab = world.prefab("Goblin")
    .set<Health>({50, 50})
    .set<Speed>({2.0f})
    .add<Enemy>();

// Инстанцирование сущностей на основе префаба
auto g1 = world.entity().is_a(GoblinPrefab).set<Position>({10, 0, 5});
auto g2 = world.entity().is_a(GoblinPrefab).set<Position>({20, 0, 10});
```

Изменение значений компонентов в префабе автоматически отражается на всех инстанцированных сущностях, что упрощает
настройку баланса и характеристик объектов.

## 3. Отношения (Relationships)

Отношения (Pairs) позволяют устанавливать типизированные связи между сущностями без хранения прямых указателей в
компонентах.

```cpp
// Установка связи (Отношение, Субъект)
sword.add<EquippedBy>(warrior); // Меч экипирован воином
chest.add<Contains>(gold);      // Сундук содержит золото
```

Механизм отношений позволяет выполнять сложные запросы к миру, например, поиск всех предметов, принадлежащих конкретному
персонажу, обеспечивая прозрачность связей в архитектуре.

## 4. Формирование поведения через компоненты

Композиция позволяет создавать объекты с уникальными свойствами путем комбинирования стандартных блоков данных.

| Тип сущности | Health | Position | Speed | PlayerTag | EnemyTag | FlyingTag |
|:-------------|:------:|:--------:|:-----:|:---------:|:--------:|:---------:|
| **Герой**    |  [x]   |   [x]    |  [x]  |    [x]    |   [ ]    |    [ ]    |
| **Стена**    |  [x]   |   [x]    |  [ ]  |    [ ]    |   [ ]    |    [ ]    |
| **Дракон**   |  [x]   |   [x]    |  [x]  |    [ ]    |   [x]    |    [x]    |

Добавление компонента (например, `FlyingTag`) мгновенно включает сущность в выборку соответствующих систем обработки,
обеспечивая высокую модульность кода.
